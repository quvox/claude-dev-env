# システム開発手順

AIと人間が協働してシステム開発する際に、守るべき開発ルールをまとめる。

消費コンテキストが必要以上に増大しないように気をつけること。Subagentを利用する場合、タスクはなるべく小さな断片になるようにすること。



## 作成すべきドキュメント

docs/の下に、開発に必要な要件定義や設計ドキュメントを配置する

* 00_idea.mdには、このプロジェクトで何を実現したいのか、一番最初のアイデアや要求を人間が記述する
* 全てはこのドキュメントから始まり、要求定義、要件定義、ユースケース、システム構造図、ADR、データモデル、API仕様、基本設計、UI/UX設計、詳細設計、テスト計画などの文書を、人間とAIが協働して作成する
* ドキュメントの冒頭に、その文書の役割を必ず記載する
* 01_xxxxのように作成した順番がわかるようにファイル名に番号をつける。もしそのようなナンバリングがされていないドキュメントがあった場合は、文書の内容から作成順序を推定してファイル名を変更する



要件変更、設計変更、機能修正、バグ対応などのたびに、ナンバリングされたドキュメントを追加しながら開発が進むため、現状の実装状況を端的に表すドキュメントが存在しない。この問題を「実装仕様ドキュメント」を作成することで解決する。

* 実装仕様ドキュメントは、docs/impl/の下に作成する
* 実装仕様ドキュメントは、要求定義、要件定義、設計、モデル、モデル、詳細設計、テスト計画は別種の文書であり、実装コードを自然言語で表現したものと考えればよい
* 実装ドキュメントは、実装に対してSingle Source of Truthでなければならない



## 実装仕様ドキュメントとは

* 要件や設計を検討した後、コードを実装する前に、**必ず**実装仕様ドキュメントを作成または更新しなければならない
* 実装仕様ドキュメントは、ソースコードと１対１に対応したドキュメントである
  * なお、小さなコードの場合は一つのドキュメントに複数のソースコードの情報を含めても良い。その場合は対応関係をドキュメント内に明記すること
  * 例えば、コードのディレクトリやサブディレクトリまで含んだ複数コードの内容を一つのドキュメントに束ねることも許容する
  * 実装コードのディレクトリ、サブディレクトリに対しては、そのディレクトリの役割や内容を説明する概要ドキュメントを作成すること

* 実装仕様には、細かな処理コードを書かかず、どのようなデータ構造、インタフェース、ロジックを実現すべきかを詳細に記載する
  * その実装仕様ドキュメントが実現しようとする機能がなぜ必要なのか（つまり要件）と、その実装仕様ドキュメントの要約を冒頭に明示する
  * また、この実装仕様ドキュメントがカバーするコードのファイルパス（ディレクトリツリー）を明示する
  * 全体を知ることなく実装できるように、実装仕様ドキュメントを記載する。
* 実装仕様ドキュメント群の全体構成を示す「実装仕様一覧」も併せて作成・更新する
  * 実装仕様一覧のファイル名は、"INDEX.md"とする
* 実装が完了したあと、その実装と実装仕様ドキュメントの間に不整合がないことを徹底的に確認する
  * 不整合とは、矛盾、考慮漏れ、記載漏れ、誤記である
  * 不整合は一切許容しない
  * 不整合があれば、実装を修正するか、ドキュメントを修正する
  * 実装仕様ドキュメントは必ず「成果物がどういうものか」を示すものでなければならない
  * 不整合がなくなるまで、何度もドキュメントを修正すること。ただし、3回以上繰り返しても不整合がなくならない場合や、解決選択肢が複数あって判断が必要な場合は、修正作業を中断して、指示を待つこと



## 実装仕様ドキュメントの管理方法

### 徹底的な確認

* 「xxxを徹底的に確認して」という指示を受けた時は、必ず10周確認作業を繰り返すこと。また1周ごとに不整合を修正して、次の周回の時には前の結果を忘却して独立試行すること
* 最終周に発見した修正点が「実装を誤らせる可能性のある矛盾や不具合」だった場合は、確認作業をさらに5周追加を繰り返し、最大30周まで繰り返す。
* 修正すべき点がなくなった場合は、10周する前に確認作業を終了する


### ドキュメントの完全性確認

* 「ドキュメントの完全性を確認して」という指示を受けた時は、以下の作業を実行すること
  - 設計ドキュメント内に不整合がないかを徹底的に確認する
  - その後、設計ドキュメントと実装ドキュメントの間に不整合がないかを徹底的に確認する
  - さらにその後に実装ドキュメントに不整合がないかを徹底的に確認する


### 変更履歴について

* ドキュメントの変更履歴はそのドキュメントには書かず、impl/histories/の下に変更履歴ファイルとして出力する
* 変更履歴ファイルは、実装仕様書ドキュメントと同じサブディレクトリ、ファイル名で保存する



### 実装後、機能変更・追加など設計変更を行う場合

* まずは、修正・追加・削除が必要な実装仕様ドキュメントを特定する
* 変更したい内容に基づいて実装仕様ドキュメントを更新する
  - 実装仕様ドキュメントは必ず「成果物がどういうものか」を示すものでなければならない
  - つまり、修正履歴のようなものは不要で、かならず最新の実装を表すもの出なければならない
* ドキュメントレベルで十分に検討し、設計不備や考慮漏れがないことを確認する
* 更新された実装仕様ドキュメントに基づいて実装を更新する
* 実装が完了したあと、その実装と実装仕様ドキュメントの間に不整合がないことを確認する
  - 不整合があれば、実装を修正するか、ドキュメントを修正する
  - 実装仕様ドキュメントは必ず「成果物がどういうものか」を示すものでなければならない
  - 不整合がなくなるまで、何度も実装を修正すること。ただし、3回以上繰り返しても不整合がなくならない場合は修正作業を中断して、指示を待つこと



### 実装はあるが、実装仕様ドキュメントが存在しない場合

* 実装を正として、実装仕様ドキュメントを逆生成する。要件定義や基本設計のドキュメントがある場合はそれも参考にすること
* 実装仕様ドキュメントは適切な粒度で作成する
  - 同じソースコードが複数の実装仕様ドキュメントに含まれないようにすること
* ドキュメントを生成したら、実装とドキュメントの内容に不整合がないかを徹底的に確認する
  - 不整合がなくなるまで、何度もドキュメントを修正すること。ただし、3回以上繰り返しても不整合がなくならない場合は修正作業を中断して、指示を待つこと

## 注意事項

- 必ず公式の最新情報、最新仕様を調べて、それを適用すること

## Web アプリの動作確認（重要）

- コンテナ内で Google Chrome が動作している。ユーザーは noVNC 経由でブラウザ画面をリアルタイムに確認できる
- chrome-devtools MCP サーバー経由で Chrome を操作すること
- **ヘッドレスブラウザを別途起動しないこと**（`chromium.launch()` 等は禁止）

### 動作確認の手順

1. 開発サーバーを起動する（`0.0.0.0` にバインドすること）
2. chrome-devtools MCP で Chrome を操作する（ページ遷移、クリック、入力、スクリーンショット等）
3. ユーザーは noVNC 画面で操作をリアルタイムに確認できる

### 注意事項
- 開発サーバーは `0.0.0.0` にバインドする（`--host 0.0.0.0` 等）
- コンテナ内の Chrome からは `localhost` で開発サーバーにアクセスできる

## Docker ネットワーク（重要）

- このシェルは Docker コンテナ `claude-dev-env` 内で動作している
- `localhost` / `127.0.0.1` では他のコンテナにアクセスできない。必ず**コンテナ名**を使うこと
  - 例: `curl http://localhost:8000` → `curl http://claude-dev-env:8000`
- 自コンテナ内のサーバーへのアクセスは `localhost` で可
- `docker ps` でコンテナ名を確認できる
- 全コンテナは Docker ネットワーク `claude-dev-net` に接続されている
