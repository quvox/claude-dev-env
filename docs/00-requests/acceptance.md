# 受入シナリオ:claude-dev-env

## シナリオ一覧

| ID | シナリオ名 | 種別(正常/異常/境界) |
|---|---|---|
| AS-1 | プロジェクトを隔離コンテナで開発開始 | 正常 |
| AS-2 | Webアプリをクライアントのブラウザで確認 | 正常 |
| AS-3 | 危険な Docker 操作が proxy に拒否される | 異常 |
| AS-4 | オーケストレーターで並列実装し、要判断だけ対応 | 正常 |
| AS-5 | 端末が壊れても実行を失わず復旧する | 異常 |

## AS-1: プロジェクトを隔離コンテナで開発開始

- **登場人物**: 社内の開発者
- **状況(前提)**: `make setup` と `make login`（OAuth）が済んでいる。あるリポジトリで開発したい。
- **一連の操作・出来事**:
  1. プロジェクトディレクトリで `claude-dev start` を実行する。
  2. コンテナが起動し、カレントディレクトリが `/workspace` にマウントされ、認証・ファイアウォールが設定される。
  3. tmux セッション内で `claude` を実行する。
- **期待する結果**: プロジェクト専用のコンテナが起動して Claude Code が動く。ホストの秘密鍵ファイルや
  Docker 生ソケットはコンテナに渡っていない。VNC ありなら起動時に noVNC URL が表示される。
- **こうなったら不合格**: 別プロジェクトの SSH 鍵がコンテナから見える／ホストの Docker を無制限に操作できる。

## AS-2: Webアプリをクライアントのブラウザで確認

- **登場人物**: 社内の開発者（Linux サーバ + SSH 運用）
- **状況(前提)**: コンテナ内で Vite アプリを `--host 0.0.0.0` で起動し、`localhost:5173` で待ち受けている。
- **一連の操作・出来事**:
  1. サーバ上で `claude-dev forward 5173` を実行する。
  2. `host:8100 → project:5173` の割当と、クライアント用の `ssh -O forward` コマンドが表示される。
  3. クライアント PC で表示された SSH トンネルを張り、`http://localhost:8100` を開く。
- **期待する結果**: クライアントのブラウザにコンテナ内 Webアプリが表示される。`claude-dev ports` で
  アクティブなフォワードを確認できる。`start` 時点ではホストにポートが公開されていない。
- **こうなったら不合格**: `start` しただけで意図せずポートがホスト公開される。

## AS-3: 危険な Docker 操作が proxy に拒否される

- **登場人物**: 社内の開発者（コンテナ内から Docker を使う）
- **状況(前提)**: Claude コンテナは `DOCKER_HOST=tcp://claude-dev-docker-proxy:2375` を使う（既定 DooD）。
- **一連の操作・出来事**:
  1. コンテナ内で `docker run -v /:/host ...`（ホストルートのバインドマウント）を試みる。
  2. リクエストが docker-proxy に届き、ボディの Binds が検査される。
- **期待する結果**: `/workspace` 配下以外のホストバインドマウント・privileged・host ネットワーク/PID 等は
  **拒否**される。`/workspace` 配下の bind は実ホストパスに書き換えて許可される（既定）。バインドを含まない
  通常の `docker run myapp` は許可される。
- **こうなったら不合格**: `/` のバインドマウントや privileged コンテナが通ってしまう。

## AS-4: オーケストレーターで並列実装し、要判断だけ対応

- **登場人物**: 社内の開発者
- **状況(前提)**: コンテナが起動している。実装したい案件がある。
- **一連の操作・出来事**:
  1. `claude-dev orchestrate` を実行する（ゴール引数なし）。ダッシュボードがホーム表示される。
  2. ブレインストーミングウィンドウの対話 Claude と一緒にゴール/仕様を固め、plan を確定して `/exit` する。
  3. 実行モードに入り、複数 worker が git worktree 上で並行実装する。人間は detach して離席する。
  4. あるタスクが要判断になり、Slack に「要判断 N 件」が来る。attach してダッシュボードのセレクタで
     ⏸ の worker を選び、その worker ウィンドウの対話で回答し `/exit` する。
- **期待する結果**: 要判断のタスクだけが待機し、他の worker は止まらず走り続ける。回答後、そのタスクが
  実行に復帰する。完了時に成果が統合され Slack に「完了」が届く。
- **こうなったら不合格**: 1件の要判断で全 worker が停止し、再開時に完了済みタスクを再実行する。

## AS-5: 端末が壊れても実行を失わず復旧する

- **登場人物**: 社内の開発者
- **状況(前提)**: `orchestrate` で実行中。マウス化け等でクライアント端末を全部閉じるしかなくなった。
- **一連の操作・出来事**:
  1. 端末（tmux クライアント）を全て閉じる。
  2. 改めて SSH で入り、`claude-dev orchestrate` を再実行する。
- **期待する結果**: tmux サーバがセッションを保持しているためコントローラは生き続けており、再 attach で
  同じ状態に戻る。コントローラが死んでいた場合も `.orchestrator/` の状態から resume し、未完了 plan は
  継続、完了済みタスクは再実行しない。plan/履歴は自動削除されない。
- **こうなったら不合格**: 端末を閉じた時点で run が失われる／再開で最初からやり直しになる。
