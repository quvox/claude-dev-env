# 要求定義書:claude-dev-env

## 1. 背景・現状の課題

チームのリソースは簡単には増やせない一方、顧客からの要望は増え続けている。単一プロジェクトを
速く終わらせることよりも、**複数プロジェクトを並列に進められること**を優先したい。今のボトルネックは
「単一案件の速度」ではなく、**人間が進行管理（オーケストレーション）を兼ねているために並列度を
上げられない**ことにある。各プロジェクトの内容を常に人間が追わねばならず、管理に時間を取られている。

同時に、エージェント（Claude Code 等）にコードを書かせ実行させると、レビュー前のコードがホストの
資産（認証情報・ホストのファイルシステム）へ到達しうるという安全上の懸念がある。開発者ごと・
プロジェクトごとに環境がばらつくと、再現性も損なわれる。

## 2. 目的・ゴール

- Claude Code を **隔離された Docker コンテナ**内で動かし、レビュー前コードがホストへ到達する経路を
  作らない。プロジェクトごとにコンテナを分離する。
- 環境一式を **Docker イメージとして配布**し、コマンド一発で立ち上げてチーム全員が同一構成を使える。
- 各プロジェクトに **AIオーケストレーターを1体**立て、人間は「検討（ブレインストーミング）」と
  「例外対応（介入）」だけに関与する。狙いは速さではなく **人間1人あたりの担当プロジェクト数を増やす**こと。

## 3. 対象ユーザー

- **社内の開発者**（本ツールの主たる利用者。日本語話者）。プロジェクトのコンテナを起動し、
  Claude Code で開発する。オーケストレーターを使って複数案件を並列に回す。
- 前提として **信頼できる社内開発用途に限定**する（信頼できないコードを扱う用途・本番に近い環境は対象外）。

## 4. 成功基準

- コマンド一発（`claude-dev start`）でプロジェクト用の隔離コンテナが起動し、その中で Claude Code が動く。
- ホストの Docker 生ソケット・秘密鍵ファイルをコンテナへ渡さずに、必要な開発（Docker 利用・SSH・
  ブラウザ確認）ができる。
- オーケストレーターを使うと、人間の関与が「介入トリガー該当時だけ」に減り、1人で複数プロジェクトを
  並行して見られる。

## 5. スコープ

### やること

- ホスト CLI（Linux: `claude-dev` / macOS: `claude-dev-mac`）によるコンテナのライフサイクル管理・
  ポートフォワード・認証・SSH 鍵転送。
- 開発ツール入りコンテナイメージ（VNC あり/なし）と、noVNC 経由のブラウザ確認。
- ホストの Docker を制限付きで使うための **Docker Socket Proxy**。
- 重い Docker 案件向けの **VM モード**（オプトイン、QEMU+virtiofs、VM 内ネイティブ Docker）。
- **AIオーケストレーター**（ブレインストーミング／実行の2モード、外部制御ループ、介入設計、
  相互レビューによる品質ゲート）。
- イメージの **GHCR 配布**（マルチアーキ）。macOS（Docker Desktop）対応。

### やらないこと（Out of Scope） ※必須

- **各エージェントの個別隔離**（microVM 等での worker ごとの隔離）。本構成は単一コンテナ同居型とし、
  隔離境界はコンテナ／ホスト間にのみ置く。個別隔離が要る用途は別構成（Docker Sandboxes 等）で検討する。
- **信頼できないコード・本番に近い環境での利用**。
- ホストの Docker 生ソケットをコンテナへ直接マウントする運用（原則採用しない）。
- オーケストレーターの **ブレインストーミング（人間の検討）の自動化**（ここは人間の価値が宿るため自動化しない）。
- v1 での「完了基準の未充足を自動でタスク化して埋める」機能（検証は助言に留める）。

## 6. 制約・前提

- 実行環境: Linux サーバ（Ubuntu 22.04+ / Debian 12+ 推奨）または macOS + Docker Desktop。Docker Engine 24+。
- 認証: Claude Pro / Max プラン（OAuth）。API キー・トークンはイメージに焼き込まず環境変数等で注入する。
- 外部 CLI（`claude` 等）と各ツールは変化が速い。採用・更新時に公式ドキュメントで最新仕様を確認する。
- OS 依存はホスト CLI に閉じる。コンテナ内資産（イメージ・entrypoint・firewall・docker-proxy）は OS 非依存に保つ。

## 7. 優先度・段階リリース

- **Must（v1 の最低限）**: 隔離コンテナ起動・認証共有・ファイアウォール・Docker Socket Proxy・
  ポートフォワード・ブラウザ確認（VNC）。
- **Should**: AIオーケストレーター（2モード・介入・相互レビュー）、VM モード、macOS 対応。
- **Could / 将来**: MCP ツールの本格連携（stdio 方式優先）、異種ベンダー worker（Codex 等）の常用、
  Docker Agent 方式の再評価。
