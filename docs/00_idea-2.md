# 要件定義：単一コンテナ内 Docker Agent によるマルチエージェント開発環境

> ステータス：ドラフト（チームレビュー用） / 対象：開発チーム
> 本書は、1つの Docker コンテナ内で Docker Agent を動かし、その配下で Claude Code・Codex などのコーディングエージェント群を連携させて開発を進める仕組みの要件を定義する。チームでの検討・改良のたたき台として用いる。

---

## 1. 背景と目的

### 1.1 背景

チームのリソースは簡単には増やせない一方、顧客からの要望は増えている。1プロジェクトの完了を速くすることよりも、**複数プロジェクトを並列に進められること**を優先したい。現状の制約は、人間がオーケストレーター（進行管理）を担うことで管理に時間を取られ、各プロジェクトの内容を常に追わねばならず、その結果として並列度が上げられないことにある。

### 1.2 目的

本仕組みは次を実現する。

- 各プロジェクトに対し、AIオーケストレーター（リードエージェント）を1体立て、人間はそれとだけ対話する
- オーケストレーターが必要に応じて Claude Code・Codex 等のコーディングエージェントを起動・連携させ、開発タスクを自律的に進める
- 人間の介入は、あらかじめ定義した「介入トリガー」に該当する時だけに限定し、人間の管理負荷を下げて並列度を上げる
- 環境一式を Docker で配布し、`docker run` 一発で立ち上げてチーム全員が同じ構成を使えるようにする

### 1.3 本書のスコープ

本書は「**1つのコンテナ内に Docker Agent とエージェント群を同居させる**」構成パターンに限定して要件を定義する。各エージェントを microVM 等で個別隔離する構成（Docker Sandboxes を用いる構成）は対象外とし、別途検討とする。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| Docker Agent | Docker 純正のオープンソースツール。エージェントを YAML で宣言的に定義し、`docker agent` コマンドで実行する CLI プラグイン。LLM のオーケストレーションループ、ツール実行、マルチエージェント委譲を担う |
| オーケストレーター（root エージェント） | Docker Agent の YAML における起点エージェント。タスクを分解し、適切なコーディングエージェントに委譲し、結果を統合する。本構成では LLM を搭載した AI オーケストレーターとする |
| ワーカー（コーディングエージェント） | 実際の開発作業を行うエージェント。本構成ではハーネス経由の Claude Code・Codex を指す |
| ハーネス | Docker Agent の機能。エージェントがモデル API を直接呼ぶ代わりに、外部コーディング CLI（claude / codex 等）に作業を委譲する仕組み。外部 CLI がコーディングループを駆動し、Docker Agent がオーケストレーション・フック・権限・配布を提供する |
| 介入トリガー | 人間の判断を仰ぐべき条件。これに該当した時だけオーケストレーターは作業を止めて人間に確認する |
| フック | Docker Agent の機構。`before_llm_call` / `after_llm_call` 等のタイミングで処理を差し込み、実行のブロックや書き換えができる。介入トリガーや承認ゲートの実装に用いる |
| DinD | Docker-in-Docker。コンテナの中でさらに Docker を動かす構成 |

---

## 3. システム概要

### 3.1 全体構成

1つの Docker コンテナ内に以下をすべて同居させる。

```
┌─ 単一 Docker コンテナ ───────────────────────┐
│                                              │
│  docker agent（ランタイム）                   │
│   ├─ root（AIオーケストレーター / LLM搭載）    │
│   │    └─ 介入トリガーを instruction に保持    │
│   ├─ claude-coder（ハーネス: claude-code）     │
│   └─ codex-coder （ハーネス: codex）           │
│                                              │
│  claude CLI / codex CLI（PATH上に配置）        │
│                                              │
│  /workspace（対象リポジトリ = ボリューム）      │
│   └─ git worktree でワーカーの作業領域を分離    │
│                                              │
│  （任意）MCP サーバー                          │
└──────────────────────────────────────────────┘
        ↑ APIキー・認証情報は環境変数等で注入
        ↓ 人間はオーケストレーターとだけ対話
```

### 3.2 動作の流れ

1. 人間がオーケストレーターにゴール（完了の定義）を伝える
2. オーケストレーターがタスクを分解し、各タスクを適切なワーカー（claude-coder / codex-coder）に `transfer_task` で委譲する
3. ワーカーが同一コンテナ内の CLI として起動し、`/workspace` のリポジトリに対して作業する
4. オーケストレーターが結果を回収し、必要なら別ワーカーにレビューさせ（品質ゲート）、改訂ループを回す
5. 介入トリガーに該当した場合のみ、人間に確認を求める。それ以外は自律で継続する
6. 完了基準を満たしたら成果を統合して完了とする

### 3.3 異種エージェント混在の意義

オーケストレーターと異なるベンダーのワーカー（Claude / Codex）を混在させることで、単一モデルでは共有しがちな盲点を相互に補い、多角的な検討による品質向上を狙う。特に「実装はワーカーA、レビューは別ベンダーのワーカーB」とすることでレビューの独立性が高まる。

## 4. 機能要件

### 4.1 オーケストレーション

- FR-1：root エージェントは LLM を搭載した AI オーケストレーターとし、与えられたゴールからタスクを動的に分解できること
- FR-2：オーケストレーターは複数のコーディングエージェントにタスクを委譲し、結果を回収・統合できること
- FR-3：タスクの委譲は、逐次（`transfer_task` で1つずつ）と並列（複数を同時ディスパッチ）の双方を選択できること
- FR-4：オーケストレーターの instruction に、第6章の介入トリガーと状況サマリの方針を組み込めること

### 4.2 コーディングエージェント（ワーカー）

- FR-5：Claude Code と Codex をハーネスとして定義し、同一コンテナ内の CLI として起動できること
- FR-6：ワーカーごとに担当領域（ディレクトリ／ファイル、または git worktree）を分離し、ファイル競合を防げること
- FR-7：ワーカーには、タスク遂行に必要な文脈（対象ファイル・仕様・制約・既決事項）を過不足なく渡せること（文脈再構築コストとコンテキスト汚染の抑制）

### 4.3 品質ゲート

- FR-8：成果物を完了とみなす前に、実装したワーカーとは別のワーカー（できれば別ベンダー）による独立レビューを通せること
- FR-9：レビューは観点を分けて行えること（例：①要件充足・動作 ②セキュリティ・エラー処理・保守性）
- FR-10：レビュー指摘を実装側に戻す改訂ループを、終了条件（重大指摘の解消／最大回数）付きで回せること

### 4.4 人間の介入

- FR-11：介入トリガー（第6章）に該当した時のみ、作業を停止して人間に確認を求められること
- FR-12：副作用のある操作（シェル実行・ファイル書き込み等）に対する承認ゲートを設けられること。開発効率を優先する場合は自動承認モードに切り替えられること
- FR-13：人間向けの状況サマリを定型フォーマットで出力し、非同期・巡回チェックで状況把握できること

### 4.5 配布・再現性

- FR-14：環境一式を Docker イメージとして構築でき、`docker run`（または compose）一発で起動できること
- FR-15：オーケストレーターとワーカーの定義（YAML）をバージョン管理でき、チームで共有・再利用できること
- FR-16：対象リポジトリはボリュームとしてマウントし、コンテナのライフサイクルと分離できること

### 4.6 MCP ツール連携（任意・将来拡張）

- FR-17：MCP ツールを利用する場合、接続方式を選択できること（stdio 方式 / Docker MCP 方式）
- FR-18：Docker MCP 方式を用いる場合、第7章のセキュリティ要件（DinD／ソケット共有）を満たすこと

---

## 5. 非機能要件

### 5.1 性能

- NFR-1：CLI 起動のオーバーヘッドは1タスクあたりの実作業時間に対して十分小さいこと。これを保つため、タスクは細切れにせず意味のあるまとまりで割り当てる（小タスクの頻繁な起動・終了を避ける）
- NFR-2：各ワーカーに渡す文脈は、コンテキストウィンドウに収まり、かつノイズで精度を落とさない範囲に絞ること。巨大リポジトリ全体を一括で渡さない

### 5.2 保守性・運用性

- NFR-3：オーケストレーター／ワーカーの定義はコード（YAML）として管理し、変更履歴を追えること
- NFR-4：介入トリガーとサマリ形式は運用しながら調整できる構造とすること（最初は緩め、運用で締める）

### 5.3 可観測性

- NFR-5：各エージェントのタスク委譲・実行結果・トークン使用量を記録できること（ハーネスは外部 CLI が使用量を報告する場合に記録される点に留意）
- NFR-6：フックによる監査ログ（プロンプトの送出前後）を取得できること

### 5.4 コスト

- NFR-7：複数ベンダーの API／サブスクリプションが並行して消費される前提でコストを見積もれること
- NFR-8：オーケストレーター自身も LLM を消費するため、調整のための消費量を把握できること

## 6. 人間の介入設計（業務要件）

本仕組みの中核。負荷削減を目的に、人間の関与を「例外時だけ」に絞る。

### 6.1 介入トリガー（これに該当する時だけ人間を呼ぶ）

1. 後戻りできない重大判断（顧客への確約、コスト発生、本番反映、データ削除、外部送信）
2. 要件の曖昧さ（推測で進めると手戻りが大きく、仮定を置くのも危険な場合）
3. 行き詰まり（同じ問題に規定回数=既定3回、別アプローチを試しても解決しない）
4. 方針の重大な分岐（全体を左右する選択で、自力で決め切る情報が足りない）
5. 前提の崩れ（当初の前提が崩れ、ゴールや納期に影響する事態を発見）

上記以外の軽微な判断は、最も妥当な仮定を置いて自律的に進め、仮定を記録する。

### 6.2 状況サマリ（非同期・巡回前提）

オーケストレーターは定型フォーマットで状況サマリを常に最新化する。ステータスが「要判断」のときだけ人間の対応を必須とし、それ以外は人間は何もしなくてよい。サマリには「進捗／ブロッカー／要判断事項／置いた仮定／次にやること」を含める。

### 6.3 実装手段

- 介入トリガーの「停止」はフック（`before_llm_call` 等）で仕組み化し、プロンプト依存にしない
- 承認ゲートは Docker Agent の副作用ツール確認機構を用いる。効率優先時は自動承認に切替可能とする

## 7. セキュリティ要件

### 7.1 隔離の前提と割り切り

- SEC-1：本構成では root・各ワーカーは同一コンテナ・同一ファイルシステムを共有し、**エージェント同士は個別には隔離されない**。隔離境界はコンテナとホストの間にのみ存在する。この割り切りを前提に、信頼できる社内開発用途に限定して用いる
- SEC-2：エージェントが実行時に生成・実行するコードはレビュー前のコードであることを認識し、ホスト資産（認証情報、ホストのファイルシステム）への到達経路を作らないこと

### 7.2 MCP / DinD に関する要件

- SEC-3：Docker MCP 方式（MCP サーバーを Docker コンテナで起動）をコンテナ内 Docker Agent から用いる場合、DinD かホストの Docker ソケット共有が必要となる
- SEC-4：ホストの Docker ソケット（`/var/run/docker.sock`）のマウントは、コンテナがホストの Docker を掌握できる重大リスクであり、原則採用しない。やむを得ず用いる場合は明示的なリスク受容とする
- SEC-5：DinD を避けたい場合は、MCP を stdio 方式（MCP サーバーをコンテナ内の一プロセスとして起動）で構成し、1コンテナに収めることを優先する
- SEC-6：最小構成（MCP を用いない）を既定とし、MCP は必要が生じた段階で stdio 方式から段階的に導入する

### 7.3 認証情報

- SEC-7：モデルプロバイダの API キー（`ANTHROPIC_API_KEY`、`OPENAI_API_KEY` 等）は環境変数等で注入し、イメージに焼き込まないこと
- SEC-8：ハーネスが用いる各 CLI 自身のログイン（トークン等）の扱いを定め、ヘッドレス環境での認証手段を確立すること
- SEC-9：認証情報は最小権限とし、不要なスコープを与えないこと

---

## 8. 前提・制約

- CON-1：ハーネスが呼ぶ外部 CLI（claude / codex）は、Docker Agent 起動前にコンテナ内の PATH 上へインストール済みであること。なければセッション開始時にエラーとなる
- CON-2：ハーネスエージェントは Docker Agent のツールセットを使用できない（外部 CLI が自前のツールを提供する）。Docker Agent のツールを併用したい場合は、ハーネスではなく通常のサブエージェント＋`transfer_task` を用いる
- CON-3：ハーネスエージェントは Docker Agent のモデルルーティング対象外（モデル選択は外部 CLI が管理する）
- CON-4：Docker Agent および各 CLI は変化が速い。採用時・更新時に各公式ドキュメントで最新の対応状況・認証方式・機能名を確認すること
- CON-5：本構成は個別エージェント隔離を提供しないため、信頼できないコードを扱う用途や本番に近い環境では Docker Sandboxes（microVM）等の別構成を検討すること

---

## 9. 段階的導入計画

1. **最小構成で1案件**：MCP なし、root＋claude-coder＋codex-coder のみ。1プロジェクトで「黙って進め、止まった時だけ呼べ」を確立し、介入頻度が十分下がることを確認する
2. **トリガーとサマリの調整**：運用しながら介入条件・サマリ形式をチューニングし、1案件あたりの人間の所要時間を実測する
3. **品質ゲートの確立**：実装ワーカーと別ベンダーのレビューワーカーによる相互レビューを定着させる
4. **横展開**：所要時間が許容内に収まってから、YAML を複製して2案件目・3案件目へ
5. **必要に応じて MCP 導入**：stdio 方式から段階的に。Docker MCP（DinD）はセキュリティ要件を満たせる場合のみ

---

## 10. 未決事項（レビューで詰める）

- オーケストレーターに用いる LLM の選定（コストと判断品質のバランス）
- 介入トリガー3の「規定回数」の最適値（既定3回でよいか）
- 承認ゲートを既定 ON とするか、自動承認を既定とするか
- ワーカーの並列実行をどこまで許すか（コスト・競合リスクとのバランス）
- 認証情報（特に各 CLI のログイン）のヘッドレス運用の具体手段
- 可観測性・監査ログの保存先と保持期間
- 本構成（同居）と Docker Sandboxes（個別隔離）の使い分け基準
