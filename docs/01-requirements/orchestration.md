# 要件定義書:オーケストレーション

## 概要

本領域は、開発環境基盤（[core.md](core.md)）上で複数のコーディングエージェントを連携させる
AIオーケストレーターの要件を定める。上流: [要求定義](../00-requests/request.md)・[決定台帳](../00-requests/decisions.md)。
狙いは速さではなく **人間1人あたりの担当プロジェクト数を増やす**こと（D-12〜D-18）。

## 用語定義

用語は [glossary.md](../00-requests/glossary.md) に従う（オーケストレーター / コントローラ / worker /
ブレインストーミングモード / 実行モード / 介入 / 介入トリガー / plan.json 等）。

## ユースケース

### UC-4:並列実装し、要判断だけ対応する（AS-4）

- **アクター:** 社内の開発者
- **目的:** 案件を自律・並列で実装させ、人間は検討と例外対応だけに関与する
- **事前条件:** コンテナが起動している
- **基本フロー:**
  1. 開発者が `claude-dev orchestrate`（ゴール引数なし）を実行する
  2. システムはコントローラを常駐起動し、ダッシュボードをホーム表示する
  3. 開発者はブレインストーミングウィンドウの対話 Claude と一緒に plan を確定し `/exit` する
  4. システムは実行モードに入り、各タスクを worker（`claude -p`、worktree 分離）へ並行ディスパッチする
  5. 開発者は detach して離席する
  6. あるタスクが要判断になると、システムは Slack に「要判断 N 件」を通知し、そのタスクだけを待機させる
  7. 開発者は attach し、ダッシュボードのセレクタで ⏸ の worker を選び、対話で回答し `/exit` する
- **代替・例外フロー:**
  - plan が実行不可（未 ready／`completion` 欠落）のとき: システムは理由を端末に明示しブレインストーミングへ戻す
  - 完了時: システムは成果を統合し Slack に「完了」を通知する
- **事後条件(成功時):** 要判断のタスクだけが待機し他 worker は継続、回答後に実行復帰、完了で成果統合
- **関連要件:** 要件12〜18

### UC-5:端末が壊れても復旧する（AS-5）

- **アクター:** 社内の開発者
- **目的:** 端末破壊・中断が起きても run を失わず同じ状態へ戻る
- **事前条件:** `orchestrate` で実行中
- **基本フロー:**
  1. 端末（tmux クライアント）が全終了する
  2. 開発者が `claude-dev orchestrate` を再実行する
  3. システムはコントローラ生存を判定し、生存なら attach、不在なら状態から resume する
- **事後条件(成功時):** 同じ状態へ復帰し、完了済みタスクは再実行されず、plan/履歴は削除されない
- **関連要件:** 要件16、要件19

### シナリオ外要件

- 自己検証（要件20）は保守的な品質確認機能であり通常の利用ジャーニーには現れない（E2E の導出元）。

## 機能要件

### 要件12:2モードのオーケストレーション

**ユーザーストーリー:** 開発者として、検討は自分で深く関与し、実装は自律・並列に任せたい。

#### 受け入れ基準

1. WHEN `claude-dev orchestrate` を実行したとき、システムはブレインストーミングモード（人間×対話Claude）から開始しなければならない
2. システムはブレインストーミング（人間の検討）を自動化してはならない
3. WHEN ブレインストーミングで plan が確定し実行が要求されたとき、システムは実行モードへ遷移し各タスクを worker へディスパッチしなければならない
4. システムはブレインストーミングと実行の引き渡し契約を実装仕様ドキュメント（`docs/`）とし、運用状態は `.orchestrator/` に保持しなければならない

### 要件13:外部制御ループとコントローラ常駐

**ユーザーストーリー:** 開発者として、暴走せず・コンテキストを汚さず・再開可能な自律実行がほしい。

#### 受け入れ基準

1. システムは継続/停止をコード（外部制御ループ）で判定しなければならない（LLM の裁量に委ねない）
2. システムはコントローラを tmux セッション `orch-<project>-main` の `dashboard` ウィンドウで常駐させ、worker/ブレインストーミングを同セッションの独立ウィンドウとして管理しなければならない
3. WHILE 反復が進む間、システムは必要な文脈だけを再構成して worker へ渡さなければならない（巨大リポジトリ全体を渡さない）

### 要件14:worker の並列実行と分離

**ユーザーストーリー:** 開発者として、複数 worker が競合せず並行作業してほしい。

#### 受け入れ基準

1. システムは各 worker を `/workspace` 内の git worktree で作業させ、ファイル競合を防がなければならない
2. システムは worker のライブ出力を当該 worker ウィンドウ（`orch-<project>-main:w-<taskID>`）で確認可能にしなければならない
3. システムは並行度（`max_workers` 等）を設定で変更可能にしなければならない

### 要件15:介入はタスク単位

**ユーザーストーリー:** 開発者として、1件の要判断で他の worker を止めたくない。

#### 受け入れ基準

1. WHEN あるタスクが介入トリガー（重大判断/曖昧さ/行き詰まり/方針分岐/前提崩れ）に該当したとき、システムはそのタスクだけを `waiting_human` にし他 worker を継続しなければならない
2. システムは判断待ちを介入キューに積み、ダッシュボードのセレクタで一覧・選択可能にしなければならない
3. WHEN 人間が回答し対話を `/exit` したとき、システムは同 worker ウィンドウでそのタスクの実行を復帰させなければならない
4. IF 軽微な判断ならば、システムは最も妥当な仮定を置いて自律継続し、仮定を状態ストアへ記録しなければならない

### 要件16:状態の保全と中断・再開

**ユーザーストーリー:** 開発者として、中断・再開で完了済み作業をやり直したくない。

#### 受け入れ基準

1. システムは起動時の自動処理で plan/状態/履歴を削除してはならない。片付けは全タスク done か `--fresh` 指定時のみとし、その場合も `history/<run_id>/` へ退避しなければならない
2. WHEN 未完了 plan が残る状態で `orchestrate` したとき、システムはその run を継続しなければならない
3. システムは `done`/`failed`/`blocked` のタスクを再開時に一切変更してはならない（途中状態のみ正規化する）
4. WHEN SIGINT/SIGTERM または `[q]` を受けたとき、システムは in-flight worker に中間コミット猶予を与え状態を保存し、終了コード 0 でクリーン終了しなければならない
5. システムは各 worker の Claude セッション ID を保存し、同一 Attempt の再開時は `--resume` で続きから再開しなければならない

### 要件17:品質ゲート（相互レビュー）

**ユーザーストーリー:** 開発者として、実装を独立した観点でレビューさせて品質を担保したい。

#### 受け入れ基準

1. システムは実装 worker と別 worker（できれば別ベンダー）による独立レビューを通さなければならない
2. システムはレビューを当該タスクの `completion` のみを基準に採点しなければならない（プランゴールで採点しない）
3. IF タスクに `completion` が未設定ならば、システムは dispatch 前に検証で弾かなければならない
4. システムはレビュー結果を構造化出力（スキーマ強制）で受け取らなければならない
5. IF 同一フォーマットエラーが 2 回続いたならば、システムは自動リトライを打ち切り介入へ回さなければならない
6. システムは改訂ループに終了条件（重大指摘の解消／最大回数 `max_review_rounds`）を設けなければならない

### 要件18:モデル選択・完了検証・Slack サマリ

**ユーザーストーリー:** 開発者として、工程に応じたモデルで動き、状況を非同期に把握したい。

#### 受け入れ基準

1. システムは工程別にモデル/effort を選ばなければならない（設計系タスク=opus/high、実装・テスト系=sonnet/high、ブレスト/介入/レビュー=opus/high）
2. WHEN 全タスクが完了したとき、システムは `completion` に対する助言的検証を行い、未充足の可能性を Slack で人間に促さなければならない（実行はブロックしない）
3. システムは状況サマリ（進捗/ブロッカー/要判断/仮定/次にやること）を Slack へ非同期に送らなければならない

### 要件19:ダッシュボード UI と日本語提示

**ユーザーストーリー:** 開発者として、今どのモードで何を待っているかが分かり、誤操作なく移動したい。

#### 受け入れ基準

1. システムはダッシュボードをイベント駆動 TUI（bubbletea）で描画し、カーソル（↑↓/jk）選択と Enter 確定でのみウィンドウ移動しなければならない（数字キー即移動・全画面再描画は行わない）
2. システムは人間に選択を求めるとき選択肢へ必ず番号を付け、番号での回答を受理しなければならない
3. システムは人間の判断・入力を求める表示（質問・選択肢・メニュー・バナー・理由・Slack 人間向け要約）をすべて日本語にしなければならない
4. システムは人間に `plan.json` 等の直接編集を促してはならない（修正は対話へ誘導する）
5. WHERE リポジトリルートに `ORCHESTRATOR.md` が存在する場合、システムはその内容をブレスト/介入指示と worker/レビュアのプロンプト先頭へ前置しなければならない

### 要件20:自己検証（バンドル題材での実走）

**ユーザーストーリー:** 保守者として、オーケストレーターの振る舞いを実題材で確認したい。

#### 受け入れ基準

1. WHEN `make orch-sample` を実行したとき、システムはバンドル題材（`examples/orch-sample`）を使い捨て作業コピー（`workspace/orch-sample`、独立 git リポジトリ）へ scaffold しなければならない（冪等。`FORCE=1` で再初期化、`SEED=1` で seed 投入）
2. システムは scaffold した題材に対し `claude-dev orchestrate` でオーケストレーターを実走させ、自己検証できなければならない（実走は E2E シナリオ E2E-4 の導出元。`make orch-sample` 自体は scaffold までで実走は含まない）
3. WHEN `make orch-sample-clean` を実行したとき、システムは題材の作業コピーを初期化しなければならない

## 非機能要件

| 分類 | 要件 |
|---|---|
| 可用性・耐障害性 | クライアント（端末）破壊でもコントローラは tmux サーバ上で生存し、`orchestrate` 再実行で復旧できる。tmux サーバ死時も `.orchestrator/` から resume する |
| 運用・保守性 | モデル/effort ポリシーは 1 箇所（`orchestrator/models.go`）で切替可能。運用状態は機械が読み書きし人間は編集しない |
| 性能 | worker は意味のあるまとまりで割り当て、小タスクの頻繁な起動・終了を避ける。文脈はコンテキストウィンドウに収まる範囲に絞る |
| 可観測性 | タスク委譲・実行結果・仮定・介入を追記型ログ（`audit.jsonl`/`assumptions.jsonl`/`interventions.jsonl`）に記録する |

## 制約(上流から継承+具体化)

- 単一コンテナ同居型・信頼できる社内開発用途に限定（core と共通、[request.md](../00-requests/request.md) §5）。
- L1 推論ループは自作せず `claude -p`／対話 Claude から借りる（D-12）。Docker Agent 方式は当面不採用。
- v1 では完了基準の未充足を自動タスク化しない（[request.md](../00-requests/request.md) §5「やらないこと」）。

## スコープ外

- ブレインストーミング（人間の検討）の自動化、異種ベンダー worker の常用（D-22 要確認）、
  Stop-hook による力技の連続走行。
