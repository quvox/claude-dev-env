# オーケストレーション設計

> **この文書の役割**: 本環境に「AI オーケストレーター」を導入し、人間の管理負荷を下げてプロジェクト並列度を上げるための設計文書。要求の出発点は [docs/00_idea-2.md](00_idea-2.md)。本書はそのアイデアから本システムに適用可能な範囲を取り出し、方式選択・全体構成・画面/プロセス像・介入設計を定める。実装の詳細仕様は別途 `docs/impl/` に起こす。

## 1. 背景と目的

チームのリソースは増やせない一方、案件は増える。ボトルネックは「単一案件の速度」ではなく、**人間がオーケストレーター（進行管理）を兼ねているため並列度が上げられない**ことにある。

本仕組みは次を実現する。

- 各プロジェクトに **AI オーケストレーターを 1 体**立て、人間はそれとだけ対話する
- オーケストレーターがコーディングエージェント（Claude Code / 将来は Codex）に作業を委譲し、**実行フェーズを自律的に進める**
- 人間の関与を「**検討（壁打ち）**」と「**例外対応（介入）**」だけに絞り、並列度を上げる

狙いは速さではなく、**人間 1 人あたりの担当プロジェクト数を増やすこと**。本書は idea-2 のスコープ（1 コンテナ同居・個別隔離なし・信頼できる社内開発用途）を踏襲し、既存の `claude-dev` 環境（1 プロジェクト＝1 コンテナ）に素直に乗せる。

## 2. 設計の中核思想

### 2.1 仕事は「壁打ち」と「実行」の 2 モードからなる

| | 壁打ちモード（検討） | 実行モード |
|---|---|---|
| 帯域 | 高帯域・同期・探索的 | 低帯域・非同期・自律 |
| 人間の関与 | 深く関与する（判断が宿る） | 手を引く（巡回チェック） |
| 成果物 | 固まった仕様/計画 | 実装・成果物 |
| 並列性 | 人間が直列（同時に深く検討できるのは 1 件） | 並列可（worker・プロジェクト） |

idea-2 は実行モードのみを設計していた。本書は**壁打ちモードを第一級の構成要素として明示**する。壁打ちは自動化しない——人間の最も価値ある貢献が宿る場所だからである。オーケストレーターが引き受けるのは「**決まったら一気に作業する**」の実行フェーズと、その並列化である。

### 2.2 オーケストレーターは「1 実体・2 モード」

「人間はオーケストレーターとだけ対話する」（単一接点）と「壁打ちは対話 Claude で行う」を両立させるため、**壁打ちは別システムではなく、同一オーケストレーターの対話モード**と位置づける。同じ実体が 2 つのモードを持つ。

```
[壁打ちモード] ──「実行開始」──▶ [実行モード] ──完了──▶ [終了]
                                  │   ▲
                          介入トリガー │   │ 解決
                                  ▼   │
                              [介入モード]
```

介入モードは、実行を一時停止して開く**集中的なミニ壁打ち**であり、壁打ちと同じ対話の仕組みを使う。ただし解決後は**実行へ復帰**する（計画全体を壁打ちからやり直すのではない）。実装上の状態名は `intervening` で、壁打ちとは別の状態として扱う（[docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

### 2.3 2 モードの境界＝実装仕様ドキュメント

壁打ちと実行の引き渡し契約は、**実装仕様/設計ドキュメント**そのものとする。CLAUDE.md は既に「コードを実装する前に、必ず実装仕様ドキュメントを作成しなければならない」と定めている。本仕組みはこの規約に乗り、**「ドキュメントが固まった → では実装」という継ぎ目にオーケストレーターを差し込む**。新しい思想を持ち込むのではなく、既存の開発ルールの後半（実装〜整合性確認）を自動化する。

```
壁打ち（人間 × 対話Claude）          ← 自動化しない
   └→ 成果物：実装仕様ドキュメント    ← 引き渡し契約（ゴール + 介入方針 + 完了基準）
        └→ オーケストレーター実行      ← ここから自動化・並列化
             └→ worker 実装 → 相互レビュー → 整合性確認
```

実行フェーズでは、この実装仕様ドキュメント（`docs/`）から構造化したタスク計画を導出して実行する。完了基準も実装仕様ドキュメントに記し、実行はそれを満たすまで進む。実行中に生じる運用状態（進捗・仮定・タスク状態・ログ）は `docs/` ではなく状態ストアに保持する（[docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

## 3. 方式選択

### 3.1 オーケストレーションの 3 層分解

| 層 | 中身 | 方針 |
|---|---|---|
| **L1 推論ループ** | tool-use ループ、コンテキスト管理、リトライ、モデルルーティング | **自作しない**。Claude Code から借りる（壁打ち＝対話 Claude、worker＝`claude -p`） |
| **L2 オーケストレーション論理** | タスク分解・委譲・レビュー改訂ループ・介入判定 | 指示書（プロンプト）＋コントローラのコードで実現 |
| **L3 配管** | worker 起動・worktree 分離・hooks・Slack サマリ・ログ | **自作する**（既存資産の拡張で安価） |

### 3.2 採用方式：自作の外部制御ループ

連続走行は「**外側の制御プロセス（コントローラ）がループを所有する**」方式で実現する。各 LLM 呼び出しは境界の決まった 1 タスクとして自然に完了し、継続性はコントローラの素直なループが次の反復を起動することで生まれる。

**却下した方式：Stop-hook による力技**（セッションの終了をフックで握り潰して走らせ続ける）。暴走・無限ループ・トークン浪費・状態の不可視化が構造的に避けられないため候補から除外する。

外部制御ループの利点：

- **暴走しない**：継続/停止は境界の決まったコードの判断（LLM の裁量ではない）
- **コンテキストを汚染しない**（idea-2 NFR-2）：反復ごとに必要な文脈だけを再構成して渡す
- **再開可能**：状態をファイルに永続化するため、落ちても巡回時に再開できる
- **L1 を再発明しない**：頭脳も worker も `claude -p`／対話 Claude をそのまま使う

### 3.3 Docker Agent との関係

idea-2 は Docker Agent を前提に書かれているが、本書では**当面採用しない**。理由：

- Docker Agent が肩代わりするのは L1＋L2 の委譲配管だが、それは Claude Code が既に提供している
- 避けられない本質作業（指示書・介入トリガー方針・サマリ形式・レビュー方針）は、どちらの方式でも人間が書く
- 新規・変化の速い依存を中核に据えると、配布の安定性（`docker run` 一発）にリスクが乗る

Docker Agent の強み（異種ベンダーの対称性・宣言的 YAML ガバナンス）が必要になった段階で再評価する（§11 段階的導入を参照）。L3 配管は方式を差し替えても再利用できる。

## 4. システム構成

### 4.1 コントローラ（外部制御ループ）

`claude-dev orchestrate` で起動する単一の常駐プロセス。L2/L3 を担う自作コンポーネントで、**実装言語は Go**とする（既存の Docker Socket Proxy が Go 製であり、ビルド・配布の footprint が揃う）。責務：

1. 端末の前景を所有し、モードに応じて前景に映すものを差し替える（`git commit` が `$EDITOR` に画面を明け渡し、終わると戻ってくるのと同じ仕組み）
2. 壁打ち/介入時は対話 Claude を子プロセスとして起動する
3. 実行時は `claude -p` worker を背景の子プロセスとして起動し、結果を回収する
4. 完了/介入トリガーを明示的なコードで判定する
5. 状態ストアを読み書きし、Slack へ状況サマリを送る

### 4.2 プロセスツリー

```
tmux session: <project>
└─ window: orchestrator
   └─ controller  (claude-dev orchestrate)    ← 単一・常駐。端末前景を所有。状態ストアを読み書き
       ├─[壁打ち/介入] claude（対話TUI）          ← 前景を一時占有する子。終了で controller に戻る
       └─[実行]        claude -p worker ×N        ← 背景の子。stdout→ログ→要約
```

対話 Claude が前景にいる間、コントローラのループは（その子を待って）停止する。すなわち**壁打ち/介入中は実行が自然に止まる**——これは狙い通りの挙動である。

### 4.3 状態ストア

壁打ちモードと実行モードの橋となる**運用状態**を、プロジェクト内のファイル群として永続化する（進捗・置いた仮定・未決事項・タスク計画・ログ）。壁打ちで固めた**仕様そのもの**は CLAUDE.md の規約どおり `docs/`（実装仕様ドキュメント）へ書き、状態ストアはそれを参照する（具体的なファイル構成は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。これにより：

- 壁打ち終了時点で仕様が書けており、実行への引き渡しに別作業が要らない
- 実行モードは長寿命の対話セッションを引きずらず、状態から必要分を再構成できる（コンテキスト健全性）
- クラッシュ後も状態から再開できる

ただし**再開するのは「中断された run」だけ**とする。起動時に永続状態を見て、実行中／介入中（`executing`/`intervening`）で中断された run のみ再開し、完了済み（`done`）・状態なし・不明な状態のときは壁打ちから**新規開始**する（完了済みの run が次回起動を即終了させたり、古い実行状態へ無言で再開して壁打ちを飛ばしたりしない）。利用者が明示的にやり直したいときは `claude-dev orchestrate --fresh` で前回の実行状態（状態・計画・worktree・作業ブランチ）を破棄して壁打ちから始め直せる（具体的な判定・操作は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

### 4.4 worker の分離

各 worker は `/workspace` 内の **git worktree** で作業し、ファイル競合を防ぐ（idea-2 FR-6）。worktree は変更の復旧路にもなる（§10 参照）。worker の `claude -p` は出力をログへ逐次ストリームし、ダッシュボードの `[d]` でライブ確認できる（§5.2）。

### 4.5 主要な統合点（実装仕様の要点）

設計上の振る舞いを実装する際の主要な接点。詳細・正本は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)。

- **指示テンプレート**：壁打ち脳・介入脳の役割（介入トリガー方針・サマリ形式・`control.json`/`plan.json` 規約）は、イメージ同梱の `wallbounce.md` / `intervene.md` を対話 `claude` に `--append-system-prompt` で注入して与える。
- **プロジェクト固有方針 `ORCHESTRATOR.md`**：リポジトリルートに置く任意のコミット対象ファイル（gitignore される `.orchestrator/` 運用状態とは別物）。存在すれば壁打ち/介入の指示と worker/レビュアのプロンプト先頭に前置され、プロジェクト固有の判断基準を与える。
- **壁打ちの引き渡し（handoff）と終了時の選択**：壁打ち脳は `control.json` に `execute`（実行へ）/`continue_wallbounce`（壁打ち継続）/`abort`（中止）を書いて終了する。`control.json` が無い・不明なときは、安全側としてコントローラが端末で **続ける/実行/終了** を明示確認する（プロンプト依存で勝手に実行しない）。
- **設定（config）**：並行度や挙動はプロジェクト/ユーザ設定で変えられる（`max_workers`・`stuck_limit`・`max_review_rounds`・`worker_model`・`merge_strategy`・`worker_permission_mode`・`reviewer_vendor`）。既定値と優先順位は 60 の「設定」を参照。
- **完了検証**：全タスク完了時、`completion`（自然言語の完了基準）に対する **助言的な `claude -p` 検証**を行い、未充足の可能性があれば Slack 通知に添えて人間の最終確認を促す（実行をブロックはしない。不足分の自動タスク化は v1 では行わない）。

## 5. 画面・プロセス構成（CLI）

すべて既存の tmux 上で動く。人間は SSH で tmux にアタッチする。

### 5.1 起動フロー（ゴールは事前に渡さない）

ゴールは壁打ちの**出力**であって入力ではない。起動するとそのまま壁打ちに入る。

```
人間: claude-dev start → claude-dev orchestrate     ← ゴール引数なし
   └ コントローラ起動 → ①壁打ちモード（前景＝対話Claude、白紙から）
   └ 一緒にゴール/仕様を固め、状態ストアへ書き留める
   └ 固まったら TUI 内で「実行開始」を選択 → ②実行へ
（任意：既にゴールが明確なら claude-dev orchestrate "<ゴール>" で壁打ちを短縮できる）
```

`claude-dev orchestrate` は、既存の `claude-dev code`（実行中コンテナの新規 tmux ウィンドウに Claude を起動。[docs/04_cli-reference.md](04_cli-reference.md)）と同系統の `claude-dev` サブコマンドで、コンテナ起動（`claude-dev start`）後にコントローラを起動する。

### 5.2 各モードの画面と入出力

**① 壁打ちモード**（前景＝対話 Claude TUI）
```
┌ tmux: <project> / orchestrator ───────────────┐
│ (interactive Claude Code TUI)                  │
│  …対話履歴…                                    │
│  ? 実装方針を選んでください                       │
│    ▸ A: …   B: …   C: …      ← select→submit 使える│
│  ▌入力フィールド                                │
└────────────────────────────────────────────────┘
 入力＝あなたが TUI にタイプ/選択（同期）
 出力＝TUI。決定は裏で状態ストア（実装仕様ドキュメント）へ書かれる
```

**② 実行モード**（前景＝コントローラのステータス表示。Claude TUI ではない）
```
┌ tmux: <project> / orchestrator ───────────────┐
│ ● 実行中  goal: …                              │
│ [3/8] worker A (claude): 実装中…  1m12s         │
│ [4/8] worker B (claude): レビュー待ち            │
│ 直近サマリ: … （Slack 送信済 12:04）             │
│ 仮定ログ 3 / 介入 0                              │
│ keys: [d]worker出力 [p]一時停止 [q]中断(再開可)  │
└────────────────────────────────────────────────┘
 人間入力＝ほぼ無し
 ・[d]：worker のライブ出力（ログ末尾）の表示をトグル
 ・[p]：新規ディスパッチの一時停止／再開（実行中 worker は継続）
 ・[q]：中断。worker を止め状態を保存して終了（done にしない）。
        次回 orchestrate で中断点から再開できる（破壊的でない）
 出力＝この画面（アタッチ時）＋ Slack サマリ（常時・非同期）
 → 離席するだけなら [q] ではなく tmux を detach する。コントローラは回り続ける
```

**③ 介入モード**（前景＝対話 Claude TUI。①と同じ画面に戻る）
```
Slack 通知:「<project> 要判断：…。attach してください」
   ↓ 人間が ssh + tmux attach
┌ tmux: <project> / orchestrator ───────────────┐
│ (interactive Claude Code TUI)                  │
│  ← コントローラが"詰まった文脈"を seed して起動    │
│  ? 認証方式が2案で決め切れません。どちらに？        │
│    ▸ OAuth …   APIキー …     ← select→submit    │
│  ▌入力フィールド                                │
└────────────────────────────────────────────────┘
 入力/出力＝TUI（同期）。回答して終了すると実行を再開（前景は②へ戻る）
```

### 5.3 ウィンドウ構成（既定：単一ウィンドウ）

対話 Claude TUI は全幅を要する（select→submit が窮屈にならないよう）ため、**対話 TUI を出すウィンドウはペイン分割しない**。

**既定（Config A）：1 ウィンドウのみ**
```
tmux session: <project>
└─ window 0: orchestrator   ← 前景がモードで切替（対話TUI ⇄ ステータス）
```
worker の中身を見たい時は、ステータス画面で `[d]` を押すと、実行中 worker のライブ出力（ログ末尾）の表示をトグルできる。worker は出力をログへストリーム書き込みするので、完了を待たずに進捗が見える。常設の worker 表示や別ウィンドウは持たない（単一ウィンドウに一本化）。

> 注：初期案にあった「Config B：worker ログ専用ウィンドウ／`--workers-window`」は**廃止**した。ダッシュボードの `[d]` ライブ表示が同じ用途を満たすため、別ウィンドウ機構と当該フラグは設けない。

### 5.4 起動〜完了の時間軸

```
T0 人間: claude-dev start → claude-dev orchestrate
   └ ①壁打ち（前景=対話Claude）。一緒にゴール/仕様を固め状態ストアへ
T1 人間が TUI 内で「実行開始」を選択
   └ ②実行へ。前景=ステータス。worker 起動。人間は detach して離席
T2 巡回: Slack のサマリを見る。異常なし→何もしない
T3 介入トリガー: コントローラが loop 停止 → Slack「要判断」
   └ 人間 ssh+tmux attach → ③前景=対話Claude（文脈 seed 済）→ select→submit で回答
   └ ②実行を再開（前景=ステータスへ）
T4 完了: 成果統合 → Slack「完了」。人間が最終確認
```

## 6. 介入設計

### 6.1 介入トリガー＝「壁打ちへ戻れ」のシグナル

実行中に判断困難へ当たると、オーケストレーターは自らループを止め、壁打ちモードへ戻る。idea-2 6.1 のトリガーを踏襲する：

1. 後戻りできない重大判断（顧客確約・コスト発生・本番反映・データ削除・外部送信）
2. 要件の曖昧さ（推測で進めると手戻りが大きく、仮定も危険）
3. 行き詰まり（同じ問題に既定 3 回、別アプローチでも解決しない）
4. 方針の重大な分岐（全体を左右する選択で自力判断の情報が足りない）
5. 前提の崩れ（当初前提が崩れ、ゴール/納期に影響）

上記以外の軽微な判断は、最も妥当な仮定を置いて自律で進め、**仮定を状態ストアに記録**する。

### 6.2 介入時の動作（確定したノブ）

- **対話 Claude はフレッシュに起動し、状態ストアから文脈を再構成する**（長寿命セッションの肥大を避ける）
- **対話 Claude を先に起動して TTY で待たせる**。人間がアタッチするとそこに質問が出ている
- 介入の受け渡しが自然になるよう、**コントローラが対話 Claude を毎回自分で起動する**（常駐セッションへ横入りする構成は採らない）

### 6.3 停止の仕組み化

介入トリガーによる「停止」はプロンプト依存にせず、コントローラのコード（および必要なら hooks）で仕組み化する（idea-2 6.3）。

## 7. 対話機能（select→submit）の扱い

Claude Code の対話機能（選択肢提示・プランモード承認・権限プロンプト等）は、**人間が同期的にいる時にだけ意味を持つ**。

| 機能 | 壁打ち/介入（前景・対話） | 実行（ヘッドレス worker） |
|---|---|---|
| テキスト入力 | ✅ 今と同じ | — 相手がいない |
| 選択肢 select→submit | ✅ 今と同じ | ✕ レンダリング先の人間がいない |
| プランモード承認 | ✅ 今と同じ | ✕ 同上 |
| 権限プロンプト | ✅ 今と同じ | ✕（bypassPermissions で自動処理） |

- 壁打ち/介入は正真正銘の対話 Claude セッションなので、**何も失われない**
- 実行中に選択が必要になっても、それは「相手が不在の時間」。軽微なら仮定して進め、重大なら**介入トリガーで壁打ちへ戻り、そこで select→submit が復活**する
- **worker は人間に直接問わない**前提で設計する。worker の責務は「完了するか、エスカレーションするか」の二択とし、人間への問いはオーケストレーター（対話モード）に集約する

## 8. 品質ゲート

- 成果を完了とみなす前に、実装した worker とは別の worker による独立レビューを通す（idea-2 FR-8）
- レビューは観点を分ける（①要件充足・動作 ②セキュリティ・エラー処理・保守性）（FR-9）
- レビュー指摘を実装側に戻す改訂ループを、終了条件（重大指摘の解消／最大回数）付きで回す（FR-10）
- フェーズ 1 は Claude のみで構成する。異種ベンダー（Codex）によるレビュー独立性の向上はフェーズ 2 で導入する（§11）

## 9. 観測性・状況サマリ

- オーケストレーターは定型サマリ（進捗／ブロッカー／要判断事項／置いた仮定／次にやること）を常に最新化し、Slack へ送る。Slack 送信は既存の通知の仕組み（`sendslackmsg.sh` が使う `SLACK_BOT_TOKEN`・チャネル設定・`chat.postMessage` 経路）を再利用する（コントローラから直接 POST するか、同等のスクリプトを呼ぶ）
- ステータスが「要判断」の時だけ人間の対応を必須とし、それ以外は巡回のみでよい（idea-2 6.2）
- 委譲・実行結果・トークン使用量・仮定・介入は、コントローラが状態ストアへ監査ログとして記録する（既存 `save_prompt.sh` はプロンプト先頭を Slack 通知用に退避する別目的のスクリプトであり、本監査ログとは別系統）
- **Slack 通知の発信源はコントローラに一本化する**。worker（`claude -p`）は Slack を送らず、結果を stdout でコントローラへ返すのみ（§7 の「worker は人間に直接問わない」と同原則）。壁打ち中の対話 Claude も、人間が同期的に在席するため Slack を送らない。これによりプロジェクトごとの通知の「声」が 1 つになる。なお従来の `claude-dev start`（オーケストレーターなし）では、これまで通り Claude Code の hook（`sendslackmsg.sh`）が通知を送る（本設計とは別系統）

## 10. セキュリティ考慮

- 既存の多層防御（コンテナ隔離・Docker Socket Proxy・ファイアウォール・認証共有）はそのまま効く。詳細は [docs/03_security.md](03_security.md)
- 既存 `settings.json` は `bypassPermissions` 前提であり、自律 worker を複数走らせると破壊的操作の影響範囲が広がる。したがって**介入トリガー・承認方針は本構成では必須の安全装置**である
- firewall/proxy はコンテナ外への被害を止めるが、`/workspace` 内の破壊は止めない。**git worktree＋コミット**を復旧路として用いる
- 本構成は個別エージェント隔離を提供しない（idea-2 SEC-1）。信頼できる社内開発用途に限定する

## 11. 段階的導入計画

idea-2 の 2 つの価値（①並列化による人間負荷削減 ②異種ベンダーによるレビュー品質）を分離して段階化する。

1. **フェーズ 1（核）**：Claude のみ。対話 Claude＝壁打ち/介入、`claude -p`＝worker。外部制御ループ・worktree 分離・介入トリガー・状況サマリ（Slack）を確立し、「黙って進め、止まった時だけ呼べ」が成立するか実測する
2. **フェーズ 2（品質ゲート）**：Codex CLI をイメージへ追加し、別ベンダーレビューを導入（コスト・認証の検討込み）
3. **フェーズ 3（必要なら）**：Docker Agent / MCP を段階導入。L3 配管は再利用する
