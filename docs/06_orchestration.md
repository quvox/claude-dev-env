# オーケストレーション設計

> **この文書の役割**: 本環境に「AI オーケストレーター」を導入し、人間の管理負荷を下げてプロジェクト並列度を上げるための設計文書。要求の出発点は [docs/00_idea-2.md](00_idea-2.md)。本書はそのアイデアから本システムに適用可能な範囲を取り出し、方式選択・全体構成・画面/プロセス像・介入設計を定める。実装の詳細仕様は別途 `docs/impl/` に起こす。

## 1. 背景と目的

チームのリソースは増やせない一方、案件は増える。ボトルネックは「単一案件の速度」ではなく、**人間がオーケストレーター（進行管理）を兼ねているため並列度が上げられない**ことにある。

本仕組みは次を実現する。

- 各プロジェクトに **AI オーケストレーターを 1 体**立て、人間はそれとだけ対話する
- オーケストレーターがコーディングエージェント（Claude Code / 将来は Codex）に作業を委譲し、**実行フェーズを自律的に進める**
- 人間の関与を「**検討（壁打ち）**」と「**例外対応（介入）**」だけに絞り、並列度を上げる

狙いは速さではなく、**人間 1 人あたりの担当プロジェクト数を増やすこと**。本書は idea-2 のスコープ（1 コンテナ同居・個別隔離なし・信頼できる社内開発用途）を踏襲し、既存の `claude-dev` 環境（1 プロジェクト＝1 コンテナ）に素直に乗せる。

## 2. 設計の中核思想

### 2.1 仕事は「壁打ち」と「実行」の 2 モードからなる

| | 壁打ちモード（検討） | 実行モード |
|---|---|---|
| 帯域 | 高帯域・同期・探索的 | 低帯域・非同期・自律 |
| 人間の関与 | 深く関与する（判断が宿る） | 手を引く（巡回チェック） |
| 成果物 | 固まった仕様/計画 | 実装・成果物 |
| 並列性 | 人間が直列（同時に深く検討できるのは 1 件） | 並列可（worker・プロジェクト） |

idea-2 は実行モードのみを設計していた。本書は**壁打ちモードを第一級の構成要素として明示**する。壁打ちは自動化しない——人間の最も価値ある貢献が宿る場所だからである。オーケストレーターが引き受けるのは「**決まったら一気に作業する**」の実行フェーズと、その並列化である。

### 2.2 オーケストレーターは「1 実体・2 モード」

「人間はオーケストレーターとだけ対話する」（単一接点）と「壁打ちは対話 Claude で行う」を両立させるため、**壁打ちは別システムではなく、同一オーケストレーターの対話モード**と位置づける。同じ実体が 2 つのモードを持つ。

```
[壁打ちモード] ──「実行開始」──▶ [実行モード] ──完了──▶ [終了]
                                  │   ▲
                  介入（タスク単位）  │   │ 回答
                                  ▼   │
                          対話 Claude（当該 worker のセッション内で判断）
```

介入は、**判断が必要になった「そのタスク 1 件」だけ**を保留し、その worker のセッションで対話 Claude を呼んで判断を仰ぐ**集中的なミニ壁打ち**である。壁打ちと同じ対話の仕組みを使い、回答後はそのタスクの実行へ復帰する（計画全体を壁打ちからやり直すのではない）。

**重要な設計原則（旧設計からの変更）**: 介入は**実行モードを離脱しない**。判断待ちのタスク以外の worker は止めない。

- 旧設計は「介入トリガー発火 → 全 worker を停止 → 独立した `intervening` 状態へ遷移 → 解決後に executing へ復帰」という**ストップ・ザ・ワールド方式**だった。これは「1 件の判断要求が並行中の全 worker を巻き込んで停止・破棄させ、再開時に大量のやり直しを生む」という構造的欠陥を持つ（[MODIFICATION.md](MODIFICATION.md) 由来の観測 + 実機検証）。
- 新設計では、介入は**タスク単位**で扱う。あるタスクが判断を要したら、そのタスクだけを `waiting_human` 状態にする。トリガーは worker の起動前または結果返却後に判定するため、**走行中の worker を強制停止する必要は基本的にない**（§6.2。in-flight を止める中間コミット猶予は Ctrl-C／`[q]` 中断の経路＝§4.3 の話で、介入とは別）。**他の worker と独立タスクの新規ディスパッチは継続**する。判断待ちは side-channel の**介入キュー**に積み、人間は巡回時にまとめて（または個別に）回答する。
- これにより、独立した最上位状態としての `intervening` は**廃止**し、介入は `executing` の内部イベントとして処理する（状態機械・キュー・worker セッションでの対話受け渡しの詳細は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

### 2.3 2 モードの境界＝実装仕様ドキュメント

壁打ちと実行の引き渡し契約は、**実装仕様/設計ドキュメント**そのものとする。CLAUDE.md は既に「コードを実装する前に、必ず実装仕様ドキュメントを作成しなければならない」と定めている。本仕組みはこの規約に乗り、**「ドキュメントが固まった → では実装」という継ぎ目にオーケストレーターを差し込む**。新しい思想を持ち込むのではなく、既存の開発ルールの後半（実装〜整合性確認）を自動化する。

```
壁打ち（人間 × 対話Claude）          ← 自動化しない
   └→ 成果物：実装仕様ドキュメント    ← 引き渡し契約（ゴール + 介入方針 + 完了基準）
        └→ オーケストレーター実行      ← ここから自動化・並列化
             └→ worker 実装 → 相互レビュー → 整合性確認
```

実行フェーズでは、この実装仕様ドキュメント（`docs/`）から構造化したタスク計画を導出して実行する。完了基準も実装仕様ドキュメントに記し、実行はそれを満たすまで進む。実行中に生じる運用状態（進捗・仮定・タスク状態・ログ）は `docs/` ではなく状態ストアに保持する（[docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

## 3. 方式選択

### 3.1 オーケストレーションの 3 層分解

| 層 | 中身 | 方針 |
|---|---|---|
| **L1 推論ループ** | tool-use ループ、コンテキスト管理、リトライ、モデルルーティング | **自作しない**。Claude Code から借りる（壁打ち＝対話 Claude、worker＝`claude -p`） |
| **L2 オーケストレーション論理** | タスク分解・委譲・レビュー改訂ループ・介入判定 | 指示書（プロンプト）＋コントローラのコードで実現 |
| **L3 配管** | worker 起動・worktree 分離・hooks・Slack サマリ・ログ | **自作する**（既存資産の拡張で安価） |

### 3.2 採用方式：自作の外部制御ループ

連続走行は「**外側の制御プロセス（コントローラ）がループを所有する**」方式で実現する。各 LLM 呼び出しは境界の決まった 1 タスクとして自然に完了し、継続性はコントローラの素直なループが次の反復を起動することで生まれる。

**却下した方式：Stop-hook による力技**（セッションの終了をフックで握り潰して走らせ続ける）。暴走・無限ループ・トークン浪費・状態の不可視化が構造的に避けられないため候補から除外する。

外部制御ループの利点：

- **暴走しない**：継続/停止は境界の決まったコードの判断（LLM の裁量ではない）
- **コンテキストを汚染しない**（idea-2 NFR-2）：反復ごとに必要な文脈だけを再構成して渡す
- **再開可能**：状態をファイルに永続化するため、落ちても巡回時に再開できる
- **L1 を再発明しない**：頭脳も worker も `claude -p`／対話 Claude をそのまま使う

### 3.3 Docker Agent との関係

idea-2 は Docker Agent を前提に書かれているが、本書では**当面採用しない**。理由：

- Docker Agent が肩代わりするのは L1＋L2 の委譲配管だが、それは Claude Code が既に提供している
- 避けられない本質作業（指示書・介入トリガー方針・サマリ形式・レビュー方針）は、どちらの方式でも人間が書く
- 新規・変化の速い依存を中核に据えると、配布の安定性（`docker run` 一発）にリスクが乗る

Docker Agent の強み（異種ベンダーの対称性・宣言的 YAML ガバナンス）が必要になった段階で再評価する（§11 段階的導入を参照）。L3 配管は方式を差し替えても再利用できる。

## 4. システム構成

### 4.1 コントローラ（外部制御ループ）＝メインセッション常駐（tmux 常駐方式）

`claude-dev orchestrate` で起動する単一の常駐プロセス。L2/L3 を担う自作コンポーネントで、**実装言語は Go**とする（既存の Docker Socket Proxy が Go 製であり、ビルド・配布の footprint が揃う）。

**重要な方針（本改訂・tmux 常駐方式）**: コントローラは**メインの tmux セッション `orch-<project>-main` の中で常駐**し、そこにダッシュボードを描画する。**「常駐」を支えるのは tmux サーバである**——人間の端末（tmux クライアント）が壊れて全終了しても、tmux サーバはセッションを保持し続けるため、**コントローラは生き続け、再 attach（`claude-dev orchestrate`）で復旧できる**。これを第一級の要件とする。tmux のマウス設定と端末状態の不整合でマウス操作がキー入力化し端末を全部閉じるしかない事態が起きても、閉じるのは*クライアント*だけで*セッション（＝コントローラ）*は残る。tmux サーバごと落ちた（再起動等）場合はコントローラも終了するが、運用状態は `.orchestrator/` に永続しているので、`claude-dev orchestrate` が新しいメインセッションでコントローラを起こし直し**状態から resume** する。旧「単一ウィンドウでコントローラが前景を所有し、壁打ち/介入で前景を差し替える」方式は**廃止**する（前景占有ゆえに端末破壊＝run 巻き添えになりやすく、壁打ち/介入が全実行を止め、状態も見えにくかった）。

> なぜ setsid の完全デーモン化ではないか：完全デーモン化すると端末を持てないため、ダッシュボード描画を別プロセスに分離し状態をファイル経由でやり取りする必要が生じ、実装が大きく複雑になる。tmux サーバを「常駐の器」として使えば、コントローラ＋ダッシュボードを 1 プロセスのまま（メモリ内状態のまま）維持でき、端末破壊耐性という本質要件も満たせる。tmux サーバ死という稀な事象は resume で吸収する。

責務：
1. 自身は `orch-<project>-main` セッション内で回り、他コンポーネント（壁打ち／各 worker／介入）を**それぞれ独立した tmux セッション**として起こし、そのライフサイクル（生成・終了・復旧）を管理する（§4.2）。
2. 壁打ち/介入は、対話 Claude を**該当セッション内で**起動し（自分の pane では起動しない＝メインのダッシュボードを奪わない）、attach クライアントをそのセッションへ切り替える。人間の `/exit` は `control.json` で受け取る（ポーリング監視）ので、コントローラは自分の前景をブロックしない。
3. 実行時は `claude -p` worker を起動して結果を回収し、各 worker セッションにライブ出力（ログ tail）を映す。
4. 完了/介入トリガーを明示的なコードで判定する。
5. 状態ストアを読み書きし、Slack へ状況サマリを送る。
6. **セッションの自動復旧**：期待するセッション（実行中 worker 分／壁打ち中なら wallbounce）が存在しなければ状態ストアから再構築する（メインは自分がいるセッション）。

### 4.2 プロセス／セッション構成（独立セッション・tmux 常駐）

```
tmux server（＝常駐の器。クライアント全終了でも生存）
  ├─ tmux session: orch-<project>-main       ← ★コントローラ本体がここで常駐＋ダッシュボード描画＝worker セレクタ（常設）
  ├─ tmux session: orch-<project>-wallbounce ← 壁打ち（対話 claude）。実行フェーズでは不要なら閉じる
  └─ tmux session: orch-<project>-w-<taskID> ← worker ごと（claude -p のライブ出力）。
        └─ 要判断時は同セッションが対話 claude（介入）へ切り替わり、/exit で worker 実行へ復帰

コントローラ（orch-…-main 内の claude-orchestrator プロセス）が、上記 wallbounce/worker
セッションを生成・終了・復旧し、対話 claude はそれらのセッション内へ投入する。
```

- `<project>` は**正規化コンテナ名**（プロジェクトごとに一意。実装では `COMPOSE_PROJECT_NAME` 相当。実装仕様 [docs/impl/60_orchestrator.md](impl/60_orchestrator.md) では `<CNAME>` と表記）。
- **worker セッションは「保持シェル」**として作る（コマンド直付けにしない）。理由：worker の `claude -p` が終了した瞬間にセッションが消えると、続く介入対話を同セッションで開けないため。コントローラが保持シェルへ `claude -p`（実行）→（要判断なら）対話 claude（介入）→再び `claude -p`（再ディスパッチ）を**順に投入**し、セッションはコマンドの生死と独立に存続する。生成タイミングは**タスクの dispatch 時、または pre-dispatch トリガー（後戻り不可の事前審査＝§6.2）で `waiting_human` になった時**（＝⏸ のタスクには必ずセッションが在り、セレクタから切り替えられる）。**タスク settle（done/failed/blocked）でコントローラが閉じる**。実装詳細は 60。
- **人間はどのセッションにも自由に attach/detach**でき、クライアントを閉じても（端末破壊でも）tmux サーバがセッションを保持するので、コントローラは生き続ける。**再 attach（`claude-dev orchestrate`）＋不足セッションの再構築**で復旧する（§5.9）。誤って `orch-<project>-main` 自体を kill した／tmux サーバごと落ちた場合はコントローラも終了するが、`claude-dev orchestrate` が新メインセッションで起こし直し状態から resume する。
- コントローラは壁打ち/介入を**自分の pane ではなく各セッション内**で動かす（`control.json` をポーリングして `/exit` を検知）ため、**壁打ち/介入中も実行（他 worker）は止まらず**、メインのダッシュボードも生きたまま。ただし個々の worker は、その worker セッションが介入対話に切り替わっている間はその 1 タスクだけ待つ。
- **マウス化け対策**：orchestrator が起こす各セッションは既定 `mouse off`（キーボード操作に一本化）とし、マウス→キー入力化による状態破壊を避ける。

### 4.3 状態ストア

壁打ちモードと実行モードの橋となる**運用状態**を、プロジェクト内のファイル群として永続化する（進捗・置いた仮定・未決事項・タスク計画・ログ）。壁打ちで固めた**仕様そのもの**は CLAUDE.md の規約どおり `docs/`（実装仕様ドキュメント）へ書き、状態ストアはそれを参照する（具体的なファイル構成は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。これにより：

- 壁打ち終了時点で仕様が書けており、実行への引き渡しに別作業が要らない（正常系。plan が実行不可＝`completion` 欠落等なら §4.5 のとおり理由を出して壁打ちへ戻す）
- 実行モードは長寿命の対話セッションを引きずらず、状態から必要分を再構成できる（コンテキスト健全性）
- クラッシュ後も状態から再開できる

**原則：運用状態（`.orchestrator/`）は機械が書く内部状態であり、人間は編集しない（UX 上の不変条件）**。`plan.json`・`control.json`・`state.json` 等は壁打ち脳／介入脳／コントローラが読み書きするものであって、**人間が手で開いて直すことは想定しない**。人間の接点は**壁打ち（対話 claude）と実行モードのキー操作だけ**。したがって、コントローラ・壁打ち脳・介入脳が出すいかなるメッセージも、**「`plan.json` を編集してください」等のファイル直接編集を人間に促してはならない**。修正が要る場合は必ず「壁打ち（対話）で詰める」よう誘導し、実際の編集は脳が行う（`wallbounce.md`／`intervene.md`／端末バナーの文言規約）。

ただし**再開するのは「中断された run」だけ**とする。起動時に永続状態を見て、実行中（`executing`）で中断された run のみ再開し、完了済み（`done`）・状態なし・不明な状態のときは壁打ちから**新規開始**する（完了済みの run が次回起動を即終了させたり、古い実行状態へ無言で再開して壁打ちを飛ばしたりしない）。利用者が明示的にやり直したいときは `claude-dev orchestrate --fresh` で前回の実行状態（状態・計画・worktree・作業ブランチ）を破棄して壁打ちから始め直せる（具体的な判定・操作は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

#### 中断・再開の堅牢性（やり直しを最小化する）

中断（Ctrl-C・プロセス kill・クラッシュ）後の再開で「完了済みタスクをもう一度実行する」「かなり前の状態からやり直す」事象を**構造的に排除**する。これは本仕組みの利用価値（人間の巡回負荷削減）を直接左右するため、第一級の要件とする。

- **完了タスクは絶対に再実行しない**: `done` になったタスクは plan.json と作業ブランチへのコミットの両方で永続化済み。再開時の正規化は `running`/`review`/`revise`/`waiting_human` といった**途中状態だけ**を対象とし、`done`/`failed`/`blocked` は一切触らない。
- **Ctrl-C を「破壊的クラッシュ」ではなく「クリーンな中断」として扱う**: SIGINT/SIGTERM を受けたら、ダッシュボードの `[q]`（中断）と**同じ経路**で、(1) in-flight worker に猶予を与えて中間コミットさせ、(2) 状態を保存し、(3) 終了コード 0 でクリーン終了する。プロセスを即座に異常終了（`log.Fatal`）させない。
- **worker の作業を捨てない（中間コミット＋セッション再開）**:
  - worker には**意味のある区切りで worktree に逐次コミット**するよう指示する（最後に一括コミットではない）。中断されてもコミット済みの部分は worktree／ブランチに保全される。
  - 各 worker の Claude **セッション ID を状態に保存**し、中断された同一 Attempt の再開時は `claude -p --resume <session-id>` で**続きから再開**する（白紙からやり直さない）。別アプローチでの再試行（新しい Attempt）は新規セッションで始める。
- **介入で peer を巻き込まない**: §2.2 の通り、介入は判断待ちタスクだけを保留する。中断・再開のたびに無関係なタスクが pending へ戻されてやり直される、という旧挙動を生まない。

これらにより、「中断 → 再開」のコストは「中断時点で未コミットだった作業」だけに限定される。worktree は変更の復旧路でもある（§10）。

### 4.4 worker の分離

各 worker は `/workspace` 内の **git worktree** で作業し、ファイル競合を防ぐ（idea-2 FR-6）。worktree は変更の復旧路にもなる（§10 参照）。worker の `claude -p` は出力をログへ逐次ストリームし、**その worker のセッション（`orch-<project>-w-<taskID>`）でライブ確認**できる（メインのセレクタから選んで切替＝§5.3）。

### 4.5 主要な統合点（実装仕様の要点）

設計上の振る舞いを実装する際の主要な接点。詳細・正本は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)。

- **指示テンプレート**：壁打ち脳・介入脳の役割（介入トリガー方針・サマリ形式・`control.json`/`plan.json` 規約）は、イメージ同梱の `wallbounce.md` / `intervene.md` を対話 `claude` に `--append-system-prompt` で注入して与える。
- **プロジェクト固有方針 `ORCHESTRATOR.md`**：リポジトリルートに置く任意のコミット対象ファイル（gitignore される `.orchestrator/` 運用状態とは別物）。存在すれば壁打ち/介入の指示と worker/レビュアのプロンプト先頭に前置され、プロジェクト固有の判断基準を与える。
- **壁打ちの引き渡し（handoff）と終了時の選択**：壁打ち脳は `control.json` に `execute`（実行へ）/`continue_wallbounce`（壁打ち継続）/`abort`（中止）を書いて終了する。`control.json` が無い・不明なときは、安全側としてコントローラが端末で **続ける/実行/終了** を明示確認する（プロンプト依存で勝手に実行しない。詳細は下記）。なお `execute` だが plan が実行不可（`ready` でない／`completion` 欠落）なときは**メニューを出さず**、理由を明示して壁打ちへ戻す（下記「実行不可」）。
  - **壁打ちの「終わり方」（アーキ上の制約と手順）**：コントローラは壁打ちセッション内で対話 `claude` を起動し（自分の pane ではなく `orch-<project>-wallbounce` 内）、`control.json` をポーリングして人間の `/exit`（＝遷移要求の書き込み）を待つ（§4.2）。したがって**壁打ちを終えるには対話 `claude` を終了する**必要がある（`/exit` または `Ctrl-D`）。これは現行方式（L1 ループを対話 Claude から借りる §3.1）に内在する制約で、自動終了はしない。円滑な流れは「壁打ち脳が plan を確定（全タスクに `completion`）→ `control.json` に `execute` を書く → 人間が `claude` を終了 → そのまま実行へ」であり、この場合は下記メニューは出ない。壁打ち脳は、実行に足る状態になったら人間に**終了操作（`/exit`）を促す**（`wallbounce.md` の役割）。
  - **終了後の選択メニュー（カーソル選択・各項目に説明）**：`control.json` が無い/不明で決定的な handoff が無いときにコントローラが出す選択は、**テキスト入力ではなく矢印キー（↑/↓）で選ぶカーソル選択式**とし、各選択肢に**番号（§5.6）と一行説明**を添える（番号キーは即確定）。（`execute` だが実行不可のときはメニューではなく下記「実行不可」経路で壁打ちへ戻す。）選択肢と意味・`control.json` の値との対応は次のとおり：
    - **1. 続ける（continue）** ＝ `control.json` の `continue_wallbounce` 相当：対話 `claude` を再度開き、要件・`plan.json` をさらに詰める（`plan.json` は保持。編集するのは壁打ち脳＝§4.3）。
    - **2. 実行（execute）** ＝ `execute` 相当：`plan.json` の各タスクを worker へ並行ディスパッチして実装を進める。**`ready` かつ全タスクに `completion` がある**ことが条件（満たさなければ実行せず理由を表示して壁打ちへ戻る＝下記「実行不可」）。
    - **3. 終了（done）** ＝ `abort` 相当：このオーケストレーション実行を終了する（run 完了）。※`[q]`／Ctrl-C の「中断（再開可・done にしない）」とは別で、こちらは run を終える。
    非 TTY（ヘッドレス）では従来どおり **続ける（continue）** に既定し、勝手に実行しない。
  - **実行不可時は「無言で戻らない」**：`execute` を要求されても plan が実行不可（`ready` でない／タスクが `completion` 未設定）なら、コントローラは**その理由を端末に明示**したうえで壁打ちへ戻す。Slack 通知だけに出して端末に何も表示しない（＝無言で `claude` が再起動したように見える）挙動は禁止。
    - **文言は「人間に `plan.json` を編集させる」ものにしない**（重要・前述 §4.3 の原則）。人間の接点は壁打ち（対話）だけで、`plan.json` は壁打ち脳が書く。表示は「壁打ちに戻って（対話で）`completion` を補ってください」のように**対話への誘導**とし、ファイル編集を促さない。
    - 理由は**次回壁打ちの壁打ち脳へ引き渡す**（壁打ちプロンプトへの前置＋監査ログ）。これにより人間が内容を中継しなくても、脳が自動で欠けた `completion` を補える。
- **設定（config）**：並行度や挙動はプロジェクト/ユーザ設定で変えられる（`max_workers`・`stuck_limit`・`max_review_rounds`・`review_format_error_limit`・`worker_grace_seconds`・`worker_model`・`merge_strategy`・`worker_permission_mode`・`reviewer_vendor`）。既定値と優先順位は 60 の「設定」を参照。
- **完了検証**：全タスク完了時、`completion`（自然言語の完了基準）に対する **助言的な `claude -p` 検証**を行い、未充足の可能性があれば Slack 通知に添えて人間の最終確認を促す（実行をブロックはしない。不足分の自動タスク化は v1 では行わない）。

## 5. 画面・プロセス構成（CLI）

すべて既存の tmux 上で動く。人間は SSH で tmux にアタッチする。

### 5.1 起動フロー（ゴールは事前に渡さない）

ゴールは壁打ちの**出力**であって入力ではない。起動するとそのまま壁打ちに入る。

```
人間: claude-dev start → claude-dev orchestrate     ← ゴール引数なし
   └ コントローラ（常駐）起動 → ①壁打ちセッション（orch-…-wallbounce＝対話Claude、白紙から）へ attach
   └ 一緒にゴール/仕様を固め、状態ストアへ書き留める
   └ 固まったら壁打ち脳が実行を記録し、対話 Claude を /exit で終了 → ②実行へ（§4.5）
（任意：既にゴールが明確なら claude-dev orchestrate "<ゴール>" で壁打ちを短縮できる）
```

`claude-dev orchestrate` は、既存の `claude-dev code`（実行中コンテナの新規 tmux ウィンドウに Claude を起動。[docs/04_cli-reference.md](04_cli-reference.md)）と同系統の `claude-dev` サブコマンドで、コンテナ起動（`claude-dev start`）後にコントローラを起動する。

### 5.2 各モードの画面と入出力

> **注（本改訂）**: 以下の①②③の「画面」は、独立セッション方式（§4.2/§5.3）では**別々の tmux セッション**に対応する：①壁打ち＝`orch-<project>-wallbounce`、②実行/ダッシュボード＝`orch-<project>-main`（worker セレクタ兼）、③介入＝**当該 worker のセッション** `orch-<project>-w-<taskID>` が対話 claude に切り替わったもの。旧「単一ウィンドウで前景を差し替える」記述（`[d]` トグル・②③が同一ウィンドウ等）はセッション分離に読み替える。

**① 壁打ちモード**（`orch-<project>-wallbounce` セッション＝対話 Claude TUI）
```
┌ tmux: orch-<project>-wallbounce ───────────────┐
│ (interactive Claude Code TUI)                  │
│  …対話履歴…                                    │
│  ? 実装方針を選んでください                       │
│    ▸ 1. …   2. …   3. …    ← 番号(§5.6)＋select→submit│
│  ▌入力フィールド                                │
└────────────────────────────────────────────────┘
 入力＝あなたが TUI にタイプ/選択（同期）
 出力＝TUI。決定は裏で状態ストア（実装仕様ドキュメント）へ書かれる
 抜け方＝対話 claude を終了（/exit・Ctrl-D）。終了後、handoff=execute かつ実行可なら実行へ／handoff 無・不明ならメニュー（続ける/実行/終了）／実行不可なら理由表示で壁打ちへ（§4.5）
```

**② 実行モード＝メインセッション（ダッシュボード＝worker セレクタ）**（`orch-<project>-main`）
```
┌ tmux: orch-<project>-main ─────────────────────┐
│ ● 実行中  goal: …                              │
│ ▸ 1. worker t3 (claude): 実装中…  1m12s         │  ← 番号でセレクト
│   2. worker t4 (claude): レビュー待ち            │
│   3. worker t5 (claude): ⏸ 要判断（認証方式）     │  ← ⏸=選ぶと介入対話へ
│ 直近サマリ: … （Slack 送信済 12:04）             │
│ 仮定ログ 3 / 要判断 1 件                         │
│ keys: [1-9]worker選択 [p]一時停止 [q]中断     │
└────────────────────────────────────────────────┘
 ・番号（数字キー）：worker を選択 → その worker セッション（orch-<project>-w-<taskID>）へ切替。
   実行中なら claude -p のライブ出力、⏸ 要判断なら③の介入対話に入る
 ・[p]：新規ディスパッチの一時停止／再開（実行中 worker は継続）
 ・[q]：run の中断（worker に中間コミット猶予→状態保存。Ctrl-C も同じ）。ただし
   コントローラ（orch-main 常駐）自体の生存・復旧は §5.9
 出力＝この画面（attach 時）＋ Slack サマリ（常時・非同期）
 → 離席は tmux detach でよい（コントローラは常駐＝§5.9）。要判断が出ても実行は止まらない
```

**③ 介入対応＝その worker のセッション内で対話**（`orch-<project>-w-<taskID>` が対話 claude に切替。**他 worker は継続**）
```
Slack 通知:「<project> 要判断 1 件：t5 認証方式…。attach しメインで選んで対応を」
   ↓ 人間が attach → メイン(②)のセレクタで ⏸ の worker(t5) を選ぶ
┌ tmux: orch-<project>-w-t5 ─────────────────────┐
│ (interactive Claude Code TUI ← この worker の介入) │
│  ? 認証方式が2案で決め切れません。どちらに？        │
│    ▸ 1. OAuth …   2. APIキー …  ← 番号(§5.6)で回答可 │
│  ▌入力フィールド                                │
└────────────────────────────────────────────────┘
 ・対話中も他 worker は各自のセッションで走り続ける
 ・回答して **対話 claude を終了（/exit・Ctrl-D）** すると、**同じ worker セッションで
   その worker が実行へ復帰**（回答済みタスクを再ディスパッチ）。メインへ戻るのは任意
 ・複数の要判断は**各 worker のセッションで個別に**対応する（キュー全体の把握はメインのセレクタ＝§5.5）
 ・途中で /exit しても安全：**記録済み（answer.md がある）回答だけが確定**し、未回答は open のまま
   残る（メインで再び選べば続きから）。「回答した」の確定は**介入脳が answer.md に記録した時点**
 ・その worker だけ抜けるのは /exit。**Ctrl-C は run 全体のクリーン中断**（再開可）なので混同しない
 ・介入脳が control.json に abort を書いて終了した場合は run を done で終了する（単なる離脱ではない）
```

> **旧設計との違い**: 旧設計はトリガー発火の瞬間に全実行をブロックして対話 Claude を自動起動していた。新設計は、要判断が出ても**実行を止めず**、人間がメインのセレクタで当該 worker を選んだ時にその worker セッションで対話する。判断待ちのタスクのみが待機し、他は進む。

### 5.3 セッション構成と worker セレクタ（独立セッション方式）

コントローラ（`orch-<project>-main` 内に常駐）が起こす tmux セッションを、コンポーネントごとに分ける（§4.2）：

- `orch-<project>-main` … メインループ/ダッシュボード。**worker セレクタ**を兼ねる（常設）。
- `orch-<project>-wallbounce` … 壁打ち（対話 claude）。
- `orch-<project>-w-<taskID>` … 各 worker（`claude -p` のライブ出力。要判断時は介入対話に切替）。

**メインセッションから worker を選ぶ**：ダッシュボード（`orch-<project>-main`）で、実行中 worker の一覧を出し **番号（数字キー）でセレクト**して選んだ worker のセッションへ切り替える（`tmux switch-client -t orch-<project>-w-<taskID>`）。要判断が出ている worker には印（⏸）を付け、選べば §5.2③ の介入対話に入る。worker セッションから戻るとメインへ戻る。

- worker のライブ出力は**その worker セッションで直接**見える（旧 `[d]` トグルは、その用途をセッション分離で置換）。
- 完了した worker のセッションは自動で閉じる。実行中の worker 分だけがセレクタに並ぶ。

**選択番号の可視化（タスク通番とは別列）**：ダッシュボードの各行はもともと**タスク通番** `[i/n]`（全 n タスク中の位置。待機中・完了も含む固定の並び）で描画する。これに対し **worker セレクタで押す番号は「選択可能な worker だけを 1 から数え直した連番」**であり、両者は一致しない（例：`[3/9]` のタスクがセレクタでは `2` 番）。この食い違いによる誤操作を防ぐため、**選択可能な worker（実行中／レビュー中／修正中／⏸ 要判断）の行だけ、行頭に選択番号を明示**する（タスク通番 `[i/n]` とは別列に、例 `‹1›`）。選択不可の行は同じ幅の空白で桁を揃える。さらに**キーヒント行に `[1-9]worker画面へ`** を出す（選択可能 worker が 1 件以上あるときのみ）。押した番号に対応する worker が無ければ無視する。

> 注：旧「単一ウィンドウ＋前景差し替え（Config A）」「worker ログ専用ウィンドウ（Config B）」はいずれも**廃止**。独立セッション＋セレクタに一本化した。

### 5.4 モードの可視化とナビゲーション（UX の第一級要件）

対話 claude の TUI が画面を占有するため、「**今どのモードにいるか**」「**どうやって抜けるか**」が分かりにくい、というのが現状の最大の UX 課題である（壁打ち・介入から戻れない／状態が見えない）。新しい状態機械を足すのではなく、**既存の遷移点に可視化と導線を差し込む**ことで解消する。

- **抜け方は全対話モードで統一＝対話 claude を終了（`/exit`・`Ctrl-D`）**。壁打ちも介入も、コントローラは対話 claude（該当セッション内の子）の終了を待って次に進む（§4.2）ため、**終了が唯一の「制御をコントローラへ返す」操作**であり自動では戻らない。これを設計上の共通規約として固定する。
- **モード遷移ごとに端末へラベル付きバナーを出す**。対話 claude は代替スクリーンで画面を占有するため、バナーは**対話 claude 起動の直前に印字**し、対話中は隠れるが `/exit` 後に scrollback として再表示される（対話中の迷子対策は下記の脳の口頭案内で担保）。少なくとも次を明示：
  - 入るモード名（**壁打ち／実行〔ダッシュボード〕／介入／終了メニュー**）。※終了メニューはメニュー自身がタイトル・各項目説明を表示するため専用バナーは不要。
  - そのモードの抜け方（対話モードなら「回答/検討が済んだら `/exit` でダッシュボードに戻る」）。
  - 例：介入に入るとき「▶ 介入モード：要判断に回答してください。済んだら `/exit` でダッシュボードへ戻ります」。
- **対話 claude の中でも迷子にしない**：`wallbounce.md`／`intervene.md` に、対話の冒頭・完了時に「これは壁打ち/介入モードである」「回答が済んだら `/exit` で戻る」旨を**人間に一言伝える**役割を持たせる（`--append-system-prompt` 経由）。あわせて §4.3 の原則どおり、脳は**人間に `plan.json` 等の編集を促さない**（対話で詰める旨のみ伝える）。
- **メイン（ダッシュボード）は常に状態を明示**：ヘッダに現在モード（実行中／一時停止）、worker 一覧、要判断の一覧（⏸ タスク名）と**番号（数字キー）で選ぶ導線**を示す（§5.2②/§5.3）。
- 実装は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)（`term.go` のバナー／メニュー、コントローラの遷移点）と同梱指示テンプレ（`wallbounce.md`／`intervene.md`）に落とす。

### 5.5 介入（要判断）キューのナビゲーション（UX）

複数の要判断が同時に開いているとき、「**いまどれに答えているのか**」「**次の件へどう移るのか**」が分かりにくい、という UX 課題を解消する。独立セッション方式では、**メインのセレクタが「どれ／次へ」のハブ**になる。

- **一覧＝メインのセレクタ**：メインセッション（§5.2②）が**開いている要判断の一覧（タスク名）を ⏸ 付きで列挙**する。人間はここで「何件・どのタスクが待っているか」を把握し、番号（数字キー）で 1 件選ぶ。
- **1 件ずつ、その worker のセッションで対応**：選んだ worker のセッションが対話 claude（介入）に切り替わり、その 1 件に回答する（§5.2③）。`/exit` で同セッションの worker 実行へ復帰し、メインへ戻って次の ⏸ を選ぶ。**各件はそれぞれの居場所（worker セッション）で対応**するので「今どの worker の何に答えているか」がセッション名で自明。
- **介入脳の口調（その 1 件について）**：`intervene.md` は、自分が担当するその 1 件について冒頭で「いま『<タスク名>』に回答中」、`answer.md` 記録後に「記録しました。`/exit` で戻れます」と明示する（全体の残件数はメインのセレクタが示すので、脳は自分の 1 件に集中）。
- **途中で切り上げ／特定の 1 件だけ**：いつ `/exit` しても記録済みの回答だけ確定し未回答は残る（§5.2③）。メインで再び選べば続きから。
- 実装は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md)（メインのセレクタ・worker セッションでの介入起動）と `intervene.md`（1 件対応の口調）に落とす。

### 5.6 選択肢は必ずナンバリングする（UX）

壁打ち・介入いずれでも、**人間に選択を求めるときは選択肢に必ず番号を付ける**（既定は `1.` `2.` `3.` …。必要なら `a` `b` `c` でも可）。人間が「**2**」のように番号だけで答えられるようにするための規約。番号無しの箇条書き（`- …`）で選択を迫るのは禁止。

- **seed する質問の整形**：要判断の質問に選択肢がある場合、`buildQuestion` の「選択肢:」ブロックは `- ` ではなく **`1. …` の連番**で出力する。
- **対話中に脳が動的に出す選択肢にも適用**：`wallbounce.md`／`intervene.md` に「選択肢を提示するときは必ず番号付きにし、**番号での回答を受理する**」規約を持たせる。
- claude Code TUI の select→submit（矢印選択）も引き続き使えるが、番号を併記して「番号を打つ／矢印で選ぶ」のどちらでも答えられるようにする。

### 5.7 人間に判断を求めるテキストは日本語で（UX）

本環境の利用者は日本語話者である。**人間の判断・入力を求める表示はすべて日本語**にする：壁打ち／介入の質問と選択肢、終了メニュー（続ける／実行／終了）、モードバナー、実行不可などの理由表示、Slack 通知の人間向け要約。

- **切り分けは「人間が読んで判断する箇所か否か」**。worker の内部作業・ログ・エージェント間のやり取り・コード/コミット/レビューの実務は**英語でよい**。
- **worker が要判断（`NeedsHuman`）を上げるときの質問文・選択肢は日本語で書く**（worker プロンプト規約）。万一英語で来ても、**介入脳（`intervene.md`）が人間へ提示する際に日本語化する**ことを保証する（必要なら和訳）。二重化で「人間向けは常に日本語」を担保。
- 実装：コントローラが出す文言（メニュー・バナー・理由・`buildQuestion` のラベル）は日本語で実装し、`wallbounce.md`／`intervene.md`／worker プロンプトに言語規約を明記する（[docs/impl/60_orchestrator.md](impl/60_orchestrator.md)）。

### 5.8 起動〜完了の時間軸

```
T0 人間: claude-dev start → claude-dev orchestrate（コントローラ常駐＋wallbounce セッションへ attach）
   └ ①壁打ち（orch-…-wallbounce）。一緒にゴール/仕様を固め状態ストアへ
T1 壁打ち脳が実行を記録（control.json=execute）→ 人間が対話 Claude を /exit で終了
   └ ②実行へ（handoff が無/不明なら §4.5 のメニュー、実行不可なら理由表示で壁打ちへ）。
     コントローラが worker セッションを起こし worker 起動。人間は detach して離席（コントローラは常駐）
T2 巡回: Slack のサマリを見る。異常なし→何もしない
T3 介入トリガー: あるタスクが要判断 → そのタスクだけ ⏸ で待機（他は継続）→ Slack「要判断 N 件」
   └ 人間 attach → メイン(orch-…-main)のセレクタで ⏸ の worker を選ぶ → ③その worker セッションで対話 → 回答
   └ /exit で同 worker セッションの実行へ復帰。この間も他 worker は走り続ける
T4 完了: 成果統合 → Slack「完了」。人間が最終確認
```

### 5.9 常駐と復旧（端末が壊れても失わない）

端末破壊（マウス→キー入力化で全端末を閉じるしかない等）でも作業を失わないための設計。

- **コントローラは `orch-<project>-main` セッション内で常駐**する。「常駐の器」は tmux サーバである：人間の tmux クライアント（端末）が壊れて全終了しても、tmux サーバは切り離された（detached）セッションを保持し続けるため、**コントローラは走り続ける**。端末を閉じることは*クライアントの detach*であって、セッション（＝コントローラ）の終了ではない。
- **セッションは状態から再構築**：コントローラは（起動時・および定期的に）期待するセッションの有無を確認し、無ければ作り直す——実行中タスクの `w-<taskID>`／壁打ち中なら `wallbounce`（`main` は自分がいるセッションなので常設）。worker セッションを誤って閉じても、その worker が実行中なら再作成して出力を再接続する（作業自体は worktree とコミットで保全済み＝§4.3）。
- **単一の復旧コマンド**：`claude-dev orchestrate`（再実行）＝「(1) `orch-<project>-main` セッションが在ればそこへ attach するだけ（コントローラはその中で生存中）。(2) 無ければ（tmux サーバ死・main を kill 等）新しい `orch-<project>-main` を作ってコントローラを起こし直し、状態ストアから resume（Phase=executing なら実行を継続）、必要なセッションを再構築してから attach する」。人間はこれ一発で、どんな壊れ方からでも同じ状態へ戻れる。作業途中終了が起きうる前提なので、**復旧は常にこの単一手順**に集約する。
- **mouse off**：orchestrator の各セッションは既定でマウス無効（キーボード操作のみ）。マウス操作がキー入力へ化けて状態を壊す事象を根本回避する。
- 失われないことの担保：ゴール/仕様は `docs/`、運用状態は `.orchestrator/`（`plan.json` 等）、実装は各 worktree のコミット。セッションやクライアントはこれらのビューに過ぎず、消えても再構築できる。

## 6. 介入設計

### 6.1 介入トリガー＝「壁打ちへ戻れ」のシグナル

実行中に判断困難へ当たると、オーケストレーターは自らループを止め、壁打ちモードへ戻る。idea-2 6.1 のトリガーを踏襲する：

1. 後戻りできない重大判断（顧客確約・コスト発生・本番反映・データ削除・外部送信）
2. 要件の曖昧さ（推測で進めると手戻りが大きく、仮定も危険）
3. 行き詰まり（同じ問題に既定 3 回、別アプローチでも解決しない）
4. 方針の重大な分岐（全体を左右する選択で自力判断の情報が足りない）
5. 前提の崩れ（当初前提が崩れ、ゴール/納期に影響）

上記以外の軽微な判断は、最も妥当な仮定を置いて自律で進め、**仮定を状態ストアに記録**する。

### 6.2 介入はタスク単位（peer を止めない）

介入の単位は **run 全体ではなくタスク 1 件**である。あるタスクがトリガーに該当したら：

1. **そのタスクだけ**を `waiting_human` 状態にする。トリガーは worker 起動前（後戻り不可の事前審査）または worker が結果を返した後に判定されるため、この時点で当該 worker は起動前か完了済みであり、走行中の worker を強制停止する必要はない（他タスクの worker には一切触れない）。
2. 要判断を**介入キュー**（状態ストアの side-channel）に積む。`waiting_human` のタスクは worker スロットを占有しない。
3. **他の worker・独立タスクの新規ディスパッチは継続**する。判断待ちのタスクに依存する後続タスクだけが自然に待つ（依存解決の通常ルール）。
4. Slack へ「要判断 N 件」を通知する。人間は巡回時に attach し、**メインのセレクタで ⏸ の worker を選んで**（各 worker のセッションで）回答する（§5.3/§5.5）。

これが旧「全停止 → 単一 `intervening` 状態」からの最大の変更点である（§2.2）。

### 6.3 介入対応時の動作（確定したノブ）

- **対話 Claude はフレッシュに起動し、状態ストアから文脈を再構成する**（長寿命セッションの肥大を避ける）。複数の要判断が開いていても、**各 worker のセッションで 1 件ずつ**対応する（1 セッションに全件 seed する旧方式は廃止。キュー全体の把握はメインのセレクタ＝§5.5）。
- **オンデマンド・その worker のセッション内で対応（独立セッション方式）**: 要判断が出てもダッシュボード／他 worker の実行は止めない。人間はメインセッションのセレクタ（§5.3）で ⏸ の worker を選ぶ → **その worker のセッションが対話 claude（介入）に切り替わり**、その場で回答する（「どの worker の何に答えているか」が居場所で自明）。対話中も他 worker は走り続ける。回答（`answer.md` + 該当タスク更新 + `control.json`）を消費したら、**同じ worker セッションで worker 実行へ復帰**（回答済みタスクを再ディスパッチ）。複数件は各 worker セッションで個別に対応でき、キュー全体の把握はメインのセレクタで行う。
- 旧「人間が `[i]` を押すと単一ウィンドウの前景に対話 Claude を出し、まとめて回答してダッシュボードへ戻る」方式は、独立セッション＋セレクタに置換した（意味は同じ＝オンデマンド・タスク単位・peer 継続）。

### 6.4 停止の仕組み化

「どのタスクを保留し、いつ人間を呼ぶか」はプロンプト依存にせず、コントローラのコード（カウンタ／ハードルール）で仕組み化する（idea-2 6.3）。停止の影響範囲は**当該タスクに限定**し、コントローラのループ自体は走り続ける。

## 7. 対話機能（select→submit）の扱い

Claude Code の対話機能（選択肢提示・プランモード承認・権限プロンプト等）は、**人間が同期的にいる時にだけ意味を持つ**。

| 機能 | 壁打ち/介入（対話・人間が同期在席） | 実行（ヘッドレス worker） |
|---|---|---|
| テキスト入力 | ✅ 今と同じ | — 相手がいない |
| 選択肢 select→submit | ✅ 今と同じ | ✕ レンダリング先の人間がいない |
| プランモード承認 | ✅ 今と同じ | ✕ 同上 |
| 権限プロンプト | ✅ 今と同じ | ✕（bypassPermissions で自動処理） |

- 壁打ち/介入は正真正銘の対話 Claude セッションなので、**何も失われない**
- 実行中に選択が必要になっても、それは「相手が不在の時間」。軽微なら仮定して進め、重大なら**介入トリガーで壁打ちへ戻り、そこで select→submit が復活**する
- **worker は人間に直接問わない**前提で設計する。worker の責務は「完了するか、エスカレーションするか」の二択とし、人間への問いはオーケストレーター（対話モード）に集約する

## 8. 品質ゲート

- 成果を完了とみなす前に、実装した worker とは別の worker による独立レビューを通す（idea-2 FR-8）
- レビューは観点を分ける（①要件充足・動作 ②セキュリティ・エラー処理・保守性）（FR-9）
- レビュー指摘を実装側に戻す改訂ループを、終了条件（重大指摘の解消／最大回数）付きで回す（FR-10）
- フェーズ 1 は Claude のみで構成する。異種ベンダー（Codex）によるレビュー独立性の向上はフェーズ 2 で導入する（§11）

### 8.1 採点基準は「タスク固有の完了基準」（プランゴールで採点しない）

実機検証（[MODIFICATION.md](MODIFICATION.md) 由来）で、**部分タスクをプラン全体のゴールで採点**してしまい、本来別タスクの責務である項目を「critical 欠落」として誤って不合格にする事象が起きた。これを構造的に防ぐ：

- **各タスクに `completion`（そのタスク単体で判定可能な完了基準）を必須化**する。担当対象・唯一の成果物パス・満たすべき構造に加え、**責務外（全体網羅・他観点・統合・後始末は別タスク）**を明示する。`completion` 未設定のタスクは実行前のプラン検証（lint）で弾く。
- **レビューは当該タスクの `completion` のみで採点**する。プラン全体のゴールや兄弟タスクの責務で減点してはならない。`completion` 空時にプランゴールへフォールバックする旧挙動は**禁止**（必須化で代替）。
- **`completion` に関わる3つの検査は役割が別**（混同しない）：(a) **実行前 lint**＝`completion` フィールドの**存在**確認（未設定を弾く・ブロック的。§4.5）、(b) **レビュー採点**＝各タスク成果物が自身の `completion` を**満たすか**（タスク単位・改訂ループ）、(c) **完了検証（§4.5）**＝全タスク完了時の**助言的**な満足度再確認（非ブロック）。本項が定めるのは主に (a)(b)。
- **`completion` が無いプランは「実行に足る＝壁打ち完了」ではない**（重要）。壁打ち脳は、**全タスクに `completion` が揃うまで `ready=true` にせず `execute` の handoff も書かない**。すなわち「完了基準が書けていないのに壁打ちの終了（実行）を提示する」のは壁打ち脳側の誤りであり、`wallbounce.md` で明確に禁止する（実行前に自己検証し、欠けていれば壁打ちを続ける）。lint はあくまで**決定論的な最後の砦**で、壁打ち脳の自己規律を代替するものではない。両者の二重化で「completion 欠落のまま実行に流れる」を防ぐ。lint が弾いた場合は理由を端末に明示し（§4.5 の handoff 節）、次回壁打ちで壁打ち脳が補完する。

### 8.2 レビュー結果は構造化出力で受ける（パース失敗を構造的に排除）

- レビュアには結果を**構造化出力（スキーマ強制）**で返させ、散文中の JSON を正規表現で拾う方式をやめる。
- **「パース不能（フォーマット違反）」と「内容としての不合格」を区別**する。フォーマット違反はリトライしても解消しないことが多いため、同一フォーマットエラーが連続したら自動リトライを打ち切り、**実装のやり直しではなくゲート不具合**として介入へ回す。
- 実装が完了しているのにゲートだけが失敗するケースを切り分けやすくするため、stuck 介入の seed に**「成果物が `completion` を満たすかの一次確認」**を添える。

### 8.3 試行（Attempt）の浪費を抑える

- review がフォーマット不一致で失敗した場合、実装は完了しているので **worker の再ディスパッチ（実作業のやり直し）はせず、レビュー工程のみ再試行**する。
- 同一原因の連続失敗を検知したら、Attempts を使い切る前に介入へエスカレーションする。

## 9. 観測性・状況サマリ

- オーケストレーターは定型サマリ（進捗／ブロッカー／要判断事項／置いた仮定／次にやること）を常に最新化し、Slack へ送る。Slack 送信は既存の通知の仕組み（`sendslackmsg.sh` が使う `SLACK_BOT_TOKEN`・チャネル設定・`chat.postMessage` 経路）を再利用する（コントローラから直接 POST するか、同等のスクリプトを呼ぶ）
- ステータスが「要判断」の時だけ人間の対応を必須とし、それ以外は巡回のみでよい（idea-2 6.2）
- 委譲・実行結果・トークン使用量・仮定・介入は、コントローラが状態ストアへ監査ログとして記録する（既存 `save_prompt.sh` はプロンプト先頭を Slack 通知用に退避する別目的のスクリプトであり、本監査ログとは別系統）
- **Slack 通知の発信源はコントローラに一本化する**。worker（`claude -p`）は Slack を送らず、結果を stdout でコントローラへ返すのみ（§7 の「worker は人間に直接問わない」と同原則）。壁打ち中の対話 Claude も、人間が同期的に在席するため Slack を送らない。これによりプロジェクトごとの通知の「声」が 1 つになる。なお従来の `claude-dev start`（オーケストレーターなし）では、これまで通り Claude Code の hook（`sendslackmsg.sh`）が通知を送る（本設計とは別系統）

## 10. セキュリティ考慮

- 既存の多層防御（コンテナ隔離・Docker Socket Proxy・ファイアウォール・認証共有）はそのまま効く。詳細は [docs/03_security.md](03_security.md)
- 既存 `settings.json` は `bypassPermissions` 前提であり、自律 worker を複数走らせると破壊的操作の影響範囲が広がる。したがって**介入トリガー・承認方針は本構成では必須の安全装置**である
- firewall/proxy はコンテナ外への被害を止めるが、`/workspace` 内の破壊は止めない。**git worktree＋コミット**を復旧路として用いる
- 本構成は個別エージェント隔離を提供しない（idea-2 SEC-1）。信頼できる社内開発用途に限定する

## 11. 段階的導入計画

idea-2 の 2 つの価値（①並列化による人間負荷削減 ②異種ベンダーによるレビュー品質）を分離して段階化する。

1. **フェーズ 1（核）**：Claude のみ。対話 Claude＝壁打ち/介入、`claude -p`＝worker。外部制御ループ・worktree 分離・介入トリガー・状況サマリ（Slack）を確立し、「黙って進め、止まった時だけ呼べ」が成立するか実測する
2. **フェーズ 2（品質ゲート）**：Codex CLI をイメージへ追加し、別ベンダーレビューを導入（コスト・認証の検討込み）
3. **フェーズ 3（必要なら）**：Docker Agent / MCP を段階導入。L3 配管は再利用する

## 12. 未決事項

- 実行モードの「次の一手」決定を、壁打ちで作った計画の実行に寄せるか、`claude -p` の頭脳呼び出しでどこまで適応的にするか
- Slack 双方向（interactive ボタン）で軽微な選択まで非同期に捌くか、また要判断キューを Slack スレッドから直接回答できるようにするか（フェーズ 2 以降の検討）
- オーケストレーター用 LLM の選定とコスト見積（idea-2 NFR-7/8）
- worker セッション再開（`--resume`）で復元できない事象（worktree とセッションの不整合等）が起きた場合のフォールバック粒度
- 介入対応（独立セッション方式：メインのセレクタ→各 worker セッションで対話）に加えて、要判断の**非同期・並行解決**（複数を同時に別々に開いて捌く）をどこまで最適化するか

> **解決済み（本改訂で確定）**:
> - 介入の単位＝**タスク単位**。peer は止めない（§2.2 / §6）。
> - 中断・再開で完了タスクを再実行しない／worker の中間コミット＋セッション再開で作業を捨てない（§4.3）。
> - 介入トリガー 3 の規定回数＝`stuck_limit` 既定 3（config 変更可）。
> - 状態ストアのファイル構成は [docs/impl/60_orchestrator.md](impl/60_orchestrator.md) で確定。
> - 行き詰まり時の「別アプローチ」自動化範囲は失敗情報付きの再ディスパッチ（同上）。
