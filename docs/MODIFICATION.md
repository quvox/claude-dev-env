本件は特定タスク固有の不具合ではなく、**分割タスク + レビューゲート**を持つ全ての開発で再現しうる構造的問題のため、方針として明文化する。

## 観測された事象

`t-sec-creds`（3ファイル限定のセキュリティ部分レビュー、読み取り専用）は成果物を正しく生成済みだったが、レビュー工程が3回連続で失敗し stuck にエスカレーションした。真因は2点:

1. **スコープの取り違え**: タスクに `completion`（タスク固有の完了基準）が未設定だったため、レビュアーが**プラン全体のゴール**（全パッケージ網羅・別観点・統合レポート作成・中間生成物削除）でこの部分タスクを採点。本来は別タスク（t-sec-exec / t-sec-fileio / t-red-a/b / t-consolidate）の担当である項目を「critical 欠落」として不合格にした。
2. **レビュー結果の機械可読性不足**: レビュアー出力が期待スキーマに合わず「no parseable ReviewResult JSON」となり review_error。リトライしても根本原因が変わらず、attempts を浪費して stuck 化した。

---

## 追記すべき方針

### 1. タスク分解時に `completion`（タスク固有の完了基準）を必須化

- プランの各タスクには、**そのタスク単体で判定可能な** `completion` を必ず付与する。
- `completion` には次を明示する:
  - 担当する対象（ファイル/ディレクトリ/観点）と**唯一の成果物パス**
  - 成果物が満たすべき構造（フォーマット、必須項目）
  - **責務外の明示**: 「全体網羅・他観点・統合・後始末は別タスクの担当であり、本タスクの完了要件に含めない」
- `completion` 未設定のタスクは dispatch 前に検証で弾く（lint）。プラン作成エージェントへの指示にも組み込む。

### 2. レビューはタスクの `completion` を基準に採点する（プランゴールで採点しない）

- レビュー工程のプロンプトに、**採点対象は当該タスクの `completion` のみ**であり、プラン全体のゴールや兄弟タスクの責務で減点してはならない、と明記する。
- `completion` が空のときプランゴールにフォールバックする現行挙動が今回の誤採点の温床。フォールバックは禁止し、`completion` 必須化（方針1）で代替する。

### 3. レビュー結果フォーマットの強制（JSON パース失敗を構造的に排除）

- レビュアーには ReviewResult を**構造化出力（スキーマ強制 / structured output）**で返させ、散文中の JSON ブロックを正規表現で拾う方式をやめる。
- パース不能（フォーマット違反）と**内容としての不合格**を明確に区別する。前者はリトライしても解消しないことが多いため、同一フォーマットエラーが2回続いたら自動リトライを打ち切り、review_error ではなく「ゲート不具合」として即介入へ回す。

### 4. stuck エスカレーション時に成果物の実在を確認させる

- review_error 起因の stuck では、エスカレーション前（または介入の seed）に「成果物がタスクの `completion` を満たしているか」の一次確認結果を添える。
- 今回のように**成果物は完成しているがゲートだけが失敗**しているケースを、人間/介入脳が即座に切り分けられるようにする。

### 5. attempts 浪費の抑制

- review_error の原因が「フォーマット不一致」で前回と同一の場合、worker の再ディスパッチ（実作業のやり直し）ではなく**レビュー工程のみ再試行**する。実作業は完了しているのに毎回やり直すのは無駄。
- 同一原因の連続失敗を検知したら attempts を消費し切る前にエスカレーションする。

## 本介入で実施済みの暫定対応（横展開の参考）

- `t-sec-creds` を `status=done` で受理（成果物がタスク記述を満たすため）。
- 全5レビュータスクに上記方針1に沿った `completion` を追記し、残り4タスクの同種誤採点を予防。

上記のうち 1・2 は今回 plan.json へ手動反映済み。3・4・5 はコントローラ実装側の変更が必要。
