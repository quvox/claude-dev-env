# AI駆動開発ドキュメント体系 — 最終設計まとめ

本書は、4階層ドキュメント駆動開発の体系(本キット)の最終仕様と、設計判断の経緯をまとめたものである。
対象は数人月〜数十人月規模のシステム開発、実装エージェントは Claude Code を想定する。

**読者は人間**(体系を深く知りたい人・メンテナ・疑問視する人)。AIの運用規範は CLAUDE.md・
commands・templates・steering にあり、AIが本書を日常的に読む必要はない。本書はそれらの
「解説」であり、規範と食い違う記述があれば規範が正(食い違いは本書のバグとして修正する)。

## 1. 体系の一言要約

**人間は「意図」だけを表明し(要求を書く・変更を説明する・質問に答える)、AIがドキュメントと
コードを前進させ、品質は機械的な検証ゲートが担保する。** ドキュメントは常に「システムの現在
あるべき姿」を表す Single Source of Truth であり、コードと一致していることが不変条件である。

## 2. 設計原則

1. **仕様ドキュメント(4階層)はSSOT。** `docs/` 直下の 00〜03 は常に現在あるべき姿を表し、
   変更時はインプレース更新する。実行計画・タスク・TODOを書かない。
2. **階層ごとに粒度が異なり、分割単位(モジュール)は設計の成果物である。** 要求を書く時点で
   分割を知っている必要はない。分割は 02-design で定義され、03-impl と実装がその単位で
   ファンアウトする。
3. **役割の三分離。** specs(現在の姿)/ histories(変更の経緯)/ tasks(AIの作業状態)。
   それぞれ性質が異なる: インプレース更新 / 追記専用 / 進行中のみ存在し完了時削除。
4. **品質ゲートは人間の承認ではなく検証。** 人間のレビューは任意(見たいときだけ見る)。
   /doc-check が唯一の検証者であり、整合性チェック一式の通過がゲートとなる。
5. **検証状態は宣言せず、証拠から導出する。** statusフィールドは存在しない。検証の証拠
   (verifiedブロック)と現在のversionの照合だけが状態を決め、上流の変更で自動的に失効する。
6. **人間の操作は定型4スキルと意図の表明のみ。** 固有名(モジュール名・slug)を覚えて
   指定する必要はなく、対象の判定はAIが行う。

## 3. ドキュメント体系

```
project-root/
├── CLAUDE.md                  # 運用ルール(常時読込。約90行のルールブック兼インデックス)
├── docs/
│   ├── 00-requests/           # 要求パッケージ(WHY+判断台帳): request.md(人間の言葉)・
│   │                          # decisions.md(決定/委任/要確認)・glossary.md・acceptance.md・examples/
│   ├── 01-requirements/       # 要件(WHAT): 最初はcore.md 1つ。大きければ業務領域で分割
│   ├── 02-design/
│   │   ├── system.md          # 全体設計。★モジュール分割定義(この体系の要)
│   │   └── <module>.md        # モジュール詳細設計(任意)
│   ├── 03-impl/
│   │   └── <module>.md        # 実装説明書: モジュールごと。実行計画ではなく実装内容の詳細記述
│   ├── histories/             # ドキュメント変更履歴(追記専用・フラット時系列)
│   │   └── YYYY-MM-DD-<slug>.md
│   ├── tasks/                 # AIの作業用タスク(進捗永続化+DoD。進行中のみ存在、完了時削除)
│   │   └── <作業slug>.md
│   ├── _steering/             # 共通ルール: product.md(概要・用語集) / tech.md(技術) / structure.md(構成・規約)
│   ├── _templates/            # 各ドキュメントのテンプレート(見出し構成と責務境界の唯一の基準)
│   │                          # _付き=仕様ドキュメント本体(00〜03等の成果物)ではなく、前提・ひな形の区別
│   └── WORKFLOW-GUIDE.md      # 運用ガイド(人間向け)
└── .claude/skills/<name>/SKILL.md   # スキル8個
```

### 各階層の役割と粒度

| 階層 | 役割 | 粒度 | 分割の根拠 |
|---|---|---|---|
| 00-requests | なぜ作るか(WHY)+先回りした判断。request.md は人間が思いの丈を書く(「やらないこと」必須)。decisions.md は判断を決定/委任/要確認に仕分けた台帳(委任=下流AIが聞かずに裁量判断してよい範囲の明示)。glossary/acceptance/examples が具体性を担う | システム全体1ディレクトリ(役割別の定型ファイル) | 機能では分割しない |
| 01-requirements | 満たすべき振る舞い(WHAT)。EARS記法の受け入れ基準+非機能6分類+ユースケース(UC-n。E2Eシナリオの導出元) | 全体1つ→業務領域で分割 | 業務構造 |
| 02-design | 実現する構造。アーキテクチャ・**モジュール分割定義**・契約・テスト戦略(3レベル方針・結合テスト担当・E2Eシナリオ一覧)・設計判断と理由 | system.md必須+詳細設計任意 | 設計判断そのもの |
| 03-impl | 実装内容の詳細な説明(ファイル構成・モジュール詳細・API・テスト対応表)。コードと常に一致 | モジュールごと+e2e.md(唯一の標準例外: E2Eシナリオ一覧に従う) | 02の分割定義に従う |

各階層は「何を決め、何を決めないか」の責務境界を持つ(例: 01はアーキテクチャに踏み込まない、
02は実装スニペットを書かないが契約は型つきで書く、03は要件・設計を蒸し返さない)。境界は
各テンプレート冒頭のコメントに明記されている。

### steeringの常時読込とその規律

steering 3ファイルは CLAUDE.md から @import され、**毎セッション必ずコンテキストに載る**
(分離はコンテキスト節約のためではなく、更新主体・記入者・変更管理が異なるためのガバナンス措置。
トークン量はCLAUDE.mdに直書きした場合と同じ)。常時読込に値するのは「全セッションで必要な前提」
(正確なコマンド文字列・命名規約・用語集)だからであり、遅延読み込みは読み忘れ事故と引き換えになる。
代償として肥大化が単調なコスト増になるため、「全セッションで必要なものだけを置き、状況依存の
詳細は02/03へ」を規律とし、月次棚卸しで守る。

### 汎用と固有の区別(キット更新の境界)

| 分類 | ファイル | キット改訂時 |
|---|---|---|
| **汎用**(方法論の一部) | CLAUDE.md / docs/_templates/ / .claude/skills/ / ガイド類(workflow-guide, onboarding, DESIGN, 各README) | 新版で上書きしてよい |
| **固有**(プロジェクトの資産) | docs/00〜03 / histories/ / tasks/ / **_steering/**(ひな形として出荷されるが初回記入以降は固有) | 絶対に触らない |

運用原則: プロジェクト固有の規約・制約は steering に書き、**テンプレート・スキルを直接改造
しない**(改造した瞬間そのファイルは固有になり、キット更新のたびに手動マージが必要になる。
方法論自体の改善はキット側に還元する)。CLAUDE.md がプロジェクト情報を含まずルールとインデックス
に徹しているのは、この境界を成立させるためでもある。

### UI設計の扱い

画面の分割・構成は構造であり、モジュール分割と同じく**02-designの成果物**。02のUIセクション
(必須。UIなしのシステムは理由を明記)に画面一覧・遷移・主要項目・状態を置き、これが画面構成の
SSOTとなる。**ピクセルレベルの見た目は文書で規定しない**: steeringのデザイン原則・トークンに
従いAIの実装に委ねる(テキストSSOTと外部デザインツールの二重化を避ける割り切り)。UI挙動は
01のEARS基準で、ルーティング・コンポーネント構成は該当モジュールの03-implで扱う。見た目の
検証は/doc-checkの対象外。

### histories と tasks

- **histories**: 1つの変更理由につき1エントリ。「変更理由/変更内容の要約/affected(更新した
  全ドキュメントのversion遷移)」のみを記録。タスク・進捗は書かない。追記専用で、確定した
  エントリは書き換えない(訂正は新エントリ)。初版(1.0.0)とPATCHのみの変更は記録不要。
- **tasks**: 役割は2つだけ——(1)セッションをまたぐ進捗の永続化(チェックボックス+進捗メモ)、
  (2)その作業の Definition of Done。人間の承認対象ではない使い捨ての足場で、完了時に削除する
  (記録はgitとhistoriesに残る)。1作業=1ファイルなので積み重ならない。

## 4. 検証メカニズム(この体系の中核)

### 証拠と導出

検証状態を表すフィールドは存在しない。/doc-check がPASS時に書き込む**証拠**と、現在のversionの
**照合**だけが状態を決める:

```yaml
version: 1.3.0                 # MAJOR.MINOR.PATCH
verified:                      # /doc-check だけが書き込める
  at: 2026-07-09
  version: 1.3.0               # 検証した時点の自分のversion
  against:
    - doc: docs/00-requests/request.md
      version: 1.0.0           # 検証した時点の上流のversion
```

**検証済みの判定(唯一の規則)**: verifiedブロックが存在し、その version と自分の現在version、
against の各 version と上流の現在version が **MAJOR.MINOR部分で一致**すること。1つでも不一致
なら失効=未検証。失効の波及は下向きのみ(上流には及ばない)。

### versionの2層意味論

- **MINOR以上**: 意味に影響する変更(要件・契約・振る舞い・構造)。自分と下流の検証を失効させる。
- **PATCH**: 意味を変えない軽微修正(誤字・表記・書式)。検証を維持し、カスケードを起こさない。
  history記録も不要。
- 迷ったらMINOR(安全側)。

編集時にやることは「versionを上げる」だけ。statusの付け替えという操作自体が存在しない。

### 増分検証

/doc-check は全ドキュメントの**フロントマターだけ**をスキャン(head/awk等で先頭ブロックのみ読む。
軽量)して状態を導出し、**本文の検証は未検証のものだけ**行う。読む範囲も限定される(対象+source
+steering、02検証時のみ01全ファイル、03検証時のみ契約と対象モジュールのコード)。
`full` 指定で全再検証(節目での実行を推奨)。

### 収束ループ

1回の検証では取り切れず、修正が新たな不整合を生むため、/doc-check は**上限10周のチェック→修正
ループ**で収束させる(修正点がなくなり次第、上限を待たず終了)。自動修正してよいのは意味保存または上流から機械的に導出できる修正のみ
(表記統一・EARS整形・フロントマター機械誤り・カバレッジ表の欠落補完・必須セクション補完)。
未規定の値の決定や、どちらが正か決められない階層間矛盾は推測せず収集し、人間に質問して停止する。
MINOR修正が入った周はhistoryに1エントリ記録。同一セッションでの修正+再検証は独立性を弱めるため、
節目にはフレッシュセッションでの `/doc-check full` を推奨する。

### 検証内容(3分類)

- **A. 縦の整合性**: 00→01のカバレッジと混入・ユースケースの網羅(全機能要件がいずれかのUCに登場)、
  01→02のカバレッジ・分割定義・steering違反・テスト戦略の充足(結合テスト担当の全契約割当、
  E2Eシナリオ一覧の全UCカバー)、02→03の分割定義との対応・契約一致・受け入れ基準→テスト割当・
  E2Eシナリオ→e2e.mdテスト割当
- **B. 03-impl⇄コードの一致**: ファイル構成・公開I/F・環境変数・テストの実在(e2e.md含む)
- **C. 品質**: 曖昧語の残存、EARS記法とテスト可能な粒度、未解決事項、フロントマター妥当性と照合

### 明示された限界

1. 「versionを正しく上げる」規律に依存する。上げない編集、意味のある変更をPATCHと申告する編集は
   失効検出をすり抜ける(厳密化するなら内容ハッシュ方式へ強化可能)。
2. 検証の相当部分はAIの判断であり、網羅的な証明ではない。「検証済み」の意味は「/doc-check の
   チェック一式がこの版の組に対して通った」である。
3. フロントマター抽出とversion照合は完全に決定的な処理なので、スクリプト化(AIの判断を挟まない
   機械実行)が信頼性強化の第一手となる。

## 5. スキル(8個)

| スキル | 役割 |
|---|---|
| `/setup` | 開始時の対話セットアップ(調査→確認方式で steering/ を生成。「未定」は正当な回答) |
| `/gen` | 次に作るべき仕様ドキュメントをAIが判定し、生成または差分更新(検証ゲートで停止) |
| `/change <変更内容>` | 変更受付: 起点階層と影響範囲を判定し、差分適用とhistory記録を対話で駆動 |
| `/implement` | 実装対象をAIが判定し、タスク作成→実装→03-impl同期→DoD実検証→タスク削除 |
| `/doc-check [module\|full]` | 唯一の検証者。増分検証し、PASSに証拠を書き込む(新セッションでの実行推奨) |
| `/doc-status` | 全ドキュメントの検証状態(導出)と進行中タスクの一覧表示 |
| `/reverse-doc <module>` | 既存コードからの仕様逆生成(ブラウンフィールド導入用) |
| `/estimate-fp` | 仕様ドキュメントからのFP算出レポート(02完了で概算=NESMA、03完了で詳細=IFPUG準拠。導出値でありドキュメントに書かない) |

スキル(8個)は体系の必須要素ではなく「手順のピン留め+コンテキスト経済性」のための最適化であり、
自然言語での依頼でもCLAUDE.mdのルールは適用される。品質を支えるのはCLAUDE.mdの原則と
テンプレートである。

実装形式は `.claude/skills/<name>/SKILL.md`(Claude Code ではカスタムスラッシュコマンドと
スキルが統合されており、`/name` で人間が起動できるほか、AIが必要と判断して自律起動することも
できる)。規律の担保はフロントマターの起動制限ではなく、**各スキル内のゲート手順(検証済み確認・
人間への合意取得・停止条件)と CLAUDE.md の規範**が担う。起動者が誰であってもゲートは省略されない。

## 6. ワークフロー

### 新規開発

```
人間が00-requests/を用意する(request.mdは自分の言葉で。「やらないこと」必須。
上流の要求定義ファクトリーで作成するのが理想。decisions.mdの委任が問い合わせを減らす)
→ /gen → /doc-check → /gen → … (01→02で分割が決まる→03を依存順に。各ゲートで検証)
→ /implement → /doc-check(実装後の03同期を再検証してループを閉じる)
→ モジュールの依存順に繰り返す
```

### 変更(要件変更・機能追加・修正)

```
/change <変更内容>
  → AIが質問(曖昧さの解消)→ 起点階層と影響範囲の提示・合意
     (起点判定: 目的・スコープ→00 / 振る舞い→01 / 構造・分割→02 / 実装詳細→03)
  → 起点から下流へ差分を提案・適用(MINOR以上上げ→検証は自動失効)、historyに記録
→ /doc-check(失効した連鎖だけが増分検証される)
→ /implement(対象はAIが判定。作業slug=変更slug)
→ /doc-check(締めの再検証)
```

/change は一発のスキルではなく対話の開始点であり、(1)曖昧さの解消、(2)影響範囲への合意、
(3)各差分への合意、の数往復が正常である。この対話が「変更が意図と合っているか」という
機械検証できない唯一の確認を担う。対話中に別の変更を思いついたら言ってよく、AIが交通整理する
(関連する変更は統合、独立した変更は別/changeを提案、誤字レベルは即修正。1変更理由=1エントリは常に維持)。

### その他のケース

| ケース | 扱い |
|---|---|
| 新モジュールが要る機能追加 | /changeが02の分割定義を変更 → /genが新03を生成 → 通常フロー |
| バグ(コードがドキュメント違反) | /change不要。数行なら直接修正(例外規定)、大きければ/implement |
| バグ(ドキュメント自体の誤り) | /change(誤りの階層が起点) |
| ドキュメントの誤字・表記 | PATCHを上げるだけ(/change不要・検証維持・history不要) |
| 既存コードに体系を導入 | /reverse-doc で変更するモジュールだけ直前に逆生成(一括逆生成はしない) |

## 7. 主要な設計判断とその経緯

議論を通じて確立・変更された判断の記録。

| 判断 | 採用した設計 | 却下した設計と理由 |
|---|---|---|
| 実装仕様書の性格 | 実装内容の詳細記述(03-impl)。コードと常に一致するSSOT | タスクリスト/実行計画 — 完了後に価値を失い、SSOTになれない。計画はtasksへ分離 |
| 実行計画の扱い | AI作業用の使い捨てドキュメント(進捗永続化+DoD)。人間の承認対象外、完了時削除 | 人間が承認する計画書 — 現実には精読されず、ゲートとして機能しない |
| histories | 全ドキュメントの変更履歴のみ(理由・要約・version遷移)。追記専用 | 計画・進捗の同居 — 履歴と作業状態は性質が異なる別もの |
| ドキュメントの構造 | 階層で切り、階層ごとの粒度でファンアウト(00=全体、01=業務領域、02=全体+詳細、03=モジュール) | 機能フォルダ縦割り(specs/<feature>/00〜03) — 分割単位は設計の成果物であり、要求時点で知り得ない |
| 品質ゲート | /doc-checkによる整合性検証。人間のレビューは任意 | 人間の承認(approved) — 現実には全文精読されず形骸化する |
| 検証状態の表現 | statusフィールドなし。証拠(verifiedブロック)+version照合による導出 | 宣言的status — 陳腐化(戻し忘れ)と自己申告の問題で信用できない |
| 軽微修正 | PATCH(照合外)で検証維持・カスケードなし | 一律version上げ — 誤字修正で全下流が失効する過剰なカスケード |
| スキルの粒度 | /gen・/implementは引数省略可、対象はAIが判定 | 階層別gen-*3個+引数必須 — 「次に何を作るか」は状態から導出でき、人間が覚える必要がない |
| 命名 | 03は"impl"、フォルダはdocs/直下、スキルはdoc-* | "spec"の多重使用(specs/フォルダ、03-spec、spec-check) — 一貫した誤読の原因だった |
| UI設計 | 02のUIセクション必須(画面一覧・遷移・項目・状態)。ピクセルレベルは規定しない | ビジュアルモックをSSOTに含める — テキストSSOTと二重化し「どちらが正か」問題を生む |
| スキルの起動権限 | 全スキルAI起動可。規律はスキル内のゲート手順とCLAUDE.md規範で担保 | disable-model-invocation で人間限定 — 対象判定までAIに委ねた本体系のAI駆動方針と不整合。ユーザー起動まで塞ぐ不具合報告もある |
| doc-checkの収束 | 上限10周のチェック→修正ループ(修正点ゼロで早期終了。自動修正は意味保存・機械導出のみ、仕様判断は人間へ) | 1回検証+手動往復 — 確率的な検証は1回で取り切れず、修正が新たな不整合を生むため収束しない |
| 規範文書の言語 | CLAUDE.md・スキル本文とも英語(トークン効率。CLAUDE.mdは常時読込で節約効果最大)。出力雛形・見出し引用・定型文・「仕様ドキュメント」等の定義用語は日本語、応答と生成物は日本語を規則で強制 | 全文日本語 — 規範は保守時にしか人間が精読せず、実行時の恒常トークンコストが毎回発生する |

## 8. 人間とAIの役割分担(最終形)

| 人間 | AI |
|---|---|
| 00-requests を用意する(request.md は自分の言葉で。decisions.md で委任範囲を宣言) | 01以降の全ドキュメントの生成・差分更新(委任範囲内は自律判断+記録) |
| /change で変更の意図を説明し、影響範囲・差分に合意する | 起点階層・影響範囲・実装対象の判定 |
| AIが止まって質問してきたら答える | 曖昧さ・矛盾の検出と、勝手に仮定せず止まること |
| (任意)見たいドキュメントを見る、修正指示、full再検証の要求 | 整合性検証と証拠の管理(/doc-check) |
| | 実装・テスト・コミット・03-impl同期・DoD実検証 |

定常操作: `(00を書く) → /gen → /doc-check → … → /implement → /doc-check`、
変更時: `/change <説明> → /doc-check → /implement → /doc-check`。

## 9. 導入手順と今後の強化候補

**導入**: キットをリポジトリのルートに配置 → steering 3ファイル(tech/structure/product)を
自分たちの内容に記入 → 小さな題材で新規開発フローを一巡 → 小さな変更で/changeフローを一巡。

**強化候補(実運用で必要になったら)**:
1. フロントマター抽出+version照合のスクリプト化(決定的処理の機械実行。信頼性強化の第一手)
2. versionの代わりに内容ハッシュ(規律依存の完全な排除)
3. doc-check の決定的チェック(曖昧語スキャン、表の空欄、パス実在)のスクリプト分離
4. フック(PostToolUse)によるテスト・lintの機械的強制

本体系のこれ以上の改善点は机上では出にくい。実プロジェクトでの一巡から得られる摩擦
(検証精度、タスク粒度、対話の往復数)が次の改訂の入力となる。
