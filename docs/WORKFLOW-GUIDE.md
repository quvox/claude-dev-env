# WORKFLOW-GUIDE.md — 運用ガイド(人間向け)

このドキュメントは人間のための運用マニュアルです。AIはこのファイルを常時読み込む必要はありません。
本書はAI向け規範(CLAUDE.md・skills・templates)の「解説」であり、食い違いがあれば規範が正です。
なお規範のうち CLAUDE.md とスキル本文はトークン効率のため英語で書かれています(AIの応答・
生成物は日本語になるよう規則で強制されています)。日本語で読みたい内容は本書がカバーします。

## 体系の考え方

**階層ごとに粒度が異なり、分割単位(モジュール)は設計の成果物である**——これがこの体系の骨格です。
要求を書く時点でモジュール分割を知っている必要はなく、分割は02-designで決まり、03-implと実装が
その単位でファン アウトします。

| 階層 | 粒度 | 分割の根拠 |
|---|---|---|
| 00-requests | システム全体で1ディレクトリ(request.md 必須+decisions.md 必須+glossary/acceptance/examples) | 機能では分割しない(役割別の定型ファイル) |
| 01-requirements | 全体1ファイル→大きければ業務領域で分割 | 業務構造(要件定義の時点で分かる) |
| 02-design | system.md(必須)+モジュール詳細設計(任意) | **分割はここで定義する(設計判断)** |
| 03-impl | モジュールごと | 02の分割定義に従う |

3種類のドキュメントの分担:

| 場所 | 役割 | 性質 | 人間の関与 |
|---|---|---|---|
| docs/(00〜03) | システムの**現在あるべき姿**(SSOT) | インプレース更新 | 見たいときだけレビュー(任意) |
| docs/histories/ | **ドキュメントの変更履歴**(仕様ドキュメント+steering) | 追記専用・フラット時系列 | 必要なとき読む |
| docs/tasks/ | **AIの作業用**(進捗永続化+DoD) | 進行中のみ存在、完了時削除 | 承認しない |

- 03-implは実行計画ではなく「実装内容の詳細説明書」。**コードと常に一致**が不変条件。
- 品質ゲートは人間の承認ではなく**検証(/doc-check)**。statusフィールドは持たず、検証状態は
  証拠(verifiedブロック=いつ・自分のどの版・上流のどの版に対して検証したか)と現在のversionの
  照合から**導出**する。自分または上流が変われば照合の不一致で**自動的に失効**する。

## 新規システム開発の手順

1. **要求定義パッケージを用意する(人間)** — `docs/00-requests/` を置く。上流の
   要求定義ファクトリー(別セッションの支援キット)で作って持ち込むのが理想。直接書く場合は
   `docs/_templates/00-requests/` から作成する。request.md はシステム全体について自分の言葉で
   (「やらないこと」は必須)。decisions.md で判断を「決定/委任/要確認」に仕分けておくと、
   以降のAIからの問い合わせが大きく減る(委任=AIが聞かずに裁量で決めてよい範囲の宣言。
   要確認=そこに来たらAIが止まって聞く)。書き上げ後の /doc-check がパッケージの形式・
   明確さ・ファイル間整合を検証して verified にする。
2. **`/gen` と `/doc-check` の交互実行** — 以降のドキュメント作成はAIが駆動する。
   /gen が現状から「次に作るべきもの」を判定して生成し(unverified)、/doc-check(新しい
   セッションで実行推奨)が検証してPASSなら verified に更新。verified になると /gen が次へ進める。
   人間の操作は「/gen → /clear → /doc-check → /clear → /gen …」の繰り返しだけで、
   ドキュメントは読んでも読まなくてもよい。
   - 01生成時: AIの質問に答える。規模が大きければ業務領域の分割案に合意する
   - 02生成時: アーキテクチャ案と**モジュール分割案**が提示される。ここが最重要の意思決定ポイント
   - 03生成時: 分割定義の依存順にモジュールごとの実装説明書(to-be)が作られる。
     全モジュール分の後、02のE2Eシナリオ一覧にシナリオがあれば 03-impl/e2e.md も作られる
3. **`/implement`** — AIが実装対象を判定して `docs/tasks/` に作業ファイルを作り、要約だけ提示して
   **そのまま実装に進む**(計画の承認待ちはしない)。完了時に03-implが実装結果に同期され、
   DoDが1項目ずつ検証され、タスクファイルは削除される。
4. モジュールの依存順に `/gen` → `/doc-check` → `/implement` → `/doc-check`(実装で03の
   versionが上がるため、コードとの一致を含む再検証で締める)を繰り返す。

**品質は /doc-check の検証で担保する。人間のレビューは任意**(見たいときに見て、修正指示や
再検証の要求ができる)。実装中にAIが「仕様に曖昧さがある」と止まって聞いてきたときだけ応答すればよい。

## 変更(機能追加・要件変更)の手順

1. **`/change <変更内容>`** — AIが起点階層と影響ドキュメントを判定して提案してくる:
   目的・スコープ→00 / 振る舞い→01(該当domain) / 構造・**モジュール分割**→02 / 実装詳細→03(該当module)
2. AIが起点から下流へ順に**差分**を提案・適用する(更新されたドキュメントは unverified に戻る)。
   経緯は `docs/histories/YYYY-MM-DD-<変更slug>.md` に記録される。
3. `/doc-check` で関係ドキュメントが verified に戻ったら **`/implement`**(対象はAIが判定)。
4. 完了後: 仕様ドキュメントは変更後の「現在の姿」、historyに経緯、タスクファイルは消えている。

新しいモジュールが必要な変更は、必ず02-designの分割定義の変更(=設計判断)を経由します。
AIが分割定義にないモジュールを勝手に作ることは禁止されています。

## 仕様ドキュメント未整備の既存コードを変更したいとき(ブラウンフィールド)

1. **`/reverse-doc <module> <対象コードのヒント>`** — AIがコードから03/02/01を逆生成する
   (00の骨子だけ提案されるので人間が書く)。system.mdがなければ分割定義の骨子から逆生成される。
2. AIが提示する「要ヒアリング事項」に答え、`/doc-check` で検証する。
3. 以降は通常の `/change` フロー。※一括逆生成はせず、変更するモジュールだけ直前に。

## レビューのコツ(見るとき用・任意)

レビューは任意だが、見るなら効率のよい箇所は以下(全文精読は不要。/doc-check の指摘一覧と
セットで見るのが最短):

| 対象 | 重点 |
|---|---|
| 要件 | 受け入れ基準が意図と一致するか / ユースケースが実際の利用の流れと一致するか / 未解決事項が残っていないか / 非機能の数値が現実的か |
| 全体設計 | **モジュール分割の単位・根拠に納得できるか**(この体系最大の意思決定) / 要件カバレッジ表に空欄がないか / E2Eシナリオ一覧が重要なユースケースを外していないか / 設計判断の「理由」 |
| 実装説明 | テスト対応表に「未検証」が残っていないか(e2e.md 含む) / 既知の制限に許容できないものがないか |
| /change の差分 | 依頼していない変更が紛れていないか / 「やらないこと」への抵触判定が妥当か |

## 検証状態の仕組み

statusフィールドは存在しない。verified は /doc-check が合格時に書き込む**合格証**であり、
検証状態は次の1規則で導出される:

**検証済み** = フロントマターに `verified` ブロックが存在し、その `version` と自分の現在の
version、`against` の各 version と上流(source)の現在の version が **MAJOR.MINOR部分で**
一致すること。1つでも不一致なら失効=未検証。

versionは MAJOR.MINOR.PATCH の3層: 意味に影響する変更(要件・契約・振る舞い・構造)は MINOR
以上を上げ、自分と下流の検証を失効させる。意味を変えない軽微修正(誤字・表記・書式)は PATCH
のみ上げ、**検証は維持される**(02の誤字修正で全03が失効するようなカスケードを防ぐ)。
迷ったら MINOR(安全側)。失効の波及は下向きのみで、上流には及ばない(03-implは最下流なので、
その編集で失効するのは03自身だけ)。

```yaml
verified:              # /doc-check だけが書き込める(PASS時)。不合格時は削除される
  at: 2026-07-09
  version: 1.2         # 検証した時点の自分の version
  against:
    - doc: docs/00-requests/request.md
      version: 1.0     # 検証した時点の上流の version
```

- 編集時にやることは「versionを上げる」だけ。付け替え作業は存在せず、照合の不一致で自動失効する。
- ゲート(/gen・/implement)も /doc-status もこの導出で判定する。
- /doc-check は**増分検証**: フロントマターの全スキャン(軽量)で状態を導出し、本文の検証は
  未検証のものだけ行う。`/doc-check full` で全再検証(節目での実行を推奨)。
- /doc-check は**上限10周のチェック→修正ループ**で収束まで自動で回る(1回打てばよい。修正点がなくなれば上限を待たず終了)。
  自動修正されるのは意味を変えない修正と機械的に導出できる補完だけで、仕様判断が必要な指摘は
  質問として返ってくる。10周で取り切れない指摘も残存として報告される。
- 限界: (a)「versionを正しく上げる」規律に依存する。上げない編集、および意味のある変更を
  PATCHと申告する編集は、失効検出をすり抜ける(厳密化するなら version の代わりに内容ハッシュへ
  強化可能)。(b) 検証の相当部分はAIの判断であり網羅的な証明ではない。検証済みの意味は
  「/doc-check のチェック一式がこの版の組に対して通った」である。

history/task に検証状態はない(historyは記録のみ、taskはファイルの有無=進行中/完了)。

## UIの扱い(割り切り)

構造・振る舞い・項目・状態(画面一覧・遷移・主要項目・loading/error等)は**テキストが正**で、
02-designのUIセクションに置く(必須。UIがないシステムは「UIなし(理由)」と明記)。
ピクセルレベルの見た目は文書で規定せず、steering/product.md のデザイン原則・トークンに従って
AIが実装時に判断する。見た目の検証(ビジュアルリグレッション)は本体系の /doc-check の対象外で、
必要ならスクリーンショット比較等の別レイヤーで担保する。

## やってはいけないこと

- **コードだけ直して03-implを放置する。** 03-impl⇄コードの一致がこの体系の生命線。
- 02-designの分割定義にないモジュールの03-implやコードを作る(唯一の例外は 03-impl/e2e.md。
  こちらは02のE2Eシナリオ一覧が根拠で、一覧にないE2Eテストを作ることが同様に禁止)。
- 仕様ドキュメントの中にタスク・TODO・チェックボックスを書く(計画はtasksの管轄)。
- historyにタスク・進捗を書く / 確定したエントリを書き換える(訂正は新エントリ)。
- versionを上げずにドキュメントを編集する、または意味のある変更をPATCHで済ませる
  (どちらも検証失効の検出をすり抜ける。最も危険な違反)。
- /doc-check 以外が verified ブロックを書き込む。

## 汎用と固有の区別・キットの更新

- **汎用(キット管理)**: CLAUDE.md / docs/_templates/ / .claude/skills/ / ガイド類。
  方法論の一部であり、キットの新版が出たらこれらを丸ごと上書きして更新する。
- **固有(プロジェクト管理)**: docs/00〜03 / histories/ / tasks/ / steering/。
  キット更新で触らない。steeringはひな形として同梱されるが、記入した時点から固有。
- 原則: プロジェクト固有の規約はsteeringへ。**テンプレート・スキルは直接改造しない**
  (改造すると固有化し、以後のキット更新で手動マージが必要になる。方法論の改善はキット側へ還元)。

## 定期メンテナンス

- **03-impl⇄コード乖離チェック(リファクタ後・月1目安):** `/doc-check` の検証項目B
- **放置タスクの確認:** `/doc-status` で、進捗メモが更新されないまま残っているタスクがないか
- **steering の棚卸し(月1目安):** tech.md のコマンド・横断的設計判断。変更したらhistoriesに記録
- **CLAUDE.md の肥大化チェック:** 200行を超えたらルールを steering やスキルに移す

## プロジェクト開始時に準備するもの

必須は4つだけ(00の執筆を除き15分程度):

- [ ] この一式をリポジトリのルートに配置した
- [ ] `git init` した(本体系はgit前提: タスク削除後の参照・変更の追跡)
- [ ] `docs/00-requests/` を用意した(上流キット製、またはテンプレートから。request.md は
  自分の言葉で・「やらないこと」必須。decisions.md の委任宣言が問い合わせを減らす)
- [ ] `/setup` を実行し、質問に答えて `docs/_steering/` を生成した
  - 「未定」は正当な回答(未定項目は02の設計対話で決まり、AIがsteeringへの反映を提案する)
  - 既存コードがあればAIが調査して候補を出すので、確認するだけでよい
  - 手動で記入したい場合はひな形を直接編集してもよい

準備不要なもの(ワークフローが作る): 01/02/03(/gen)、histories・tasksの中身、srcディレクトリ。

推奨: 本番プロジェクトの前に、小さめの題材で「新規開発」フローと「/change」フローを一巡しておく。
