---
title: 3 段ネストのシェルクォートを避け、設定ファイル生成は最外・root 部で行う
summary: docker run -c '...' → su -c "..." → echo '...' の 3 段ネストはクォート破綻とブレース展開でスクリプトを分裂させ不正 JSON を生む。生成を最外の root 実行部へ出しネストを 1 段減らす。検証は docker inspect の Cmd で行う
---

## 状況

`claude-dev login` が一時コンテナ内に作る `~/.claude/settings.json` が
`permissions:{defaultMode:bypassPermissions}` という不正 JSON になり、Claude Code が設定破損を
報告した（macOS で顕在化、Linux も同一コード）。

根本原因は 3 段ネストのクォート破綻:

1. `docker run ... -c '...'` のコンテナ内スクリプトはホスト側シングルクォートで括られている
2. その内側 `su -c "..."` 内の `echo '{\"permissions\":...}'` の最初の `'` が外側の引用を閉じ、
   JSON がホストシェルに裸で露出
3. ホストシェルが `\"`→`"` を消費し、トップレベル `{A,B}` を**ブレース展開**して `docker run` の
   `-c` 引数が 2 つのスクリプトに分裂。実行された側が不正テキストを書き込む
   （分裂は `docker inspect` の `Cmd` で確認）。

## 判断

- **`settings.json` 生成を `su -c "..."` の内側から、`su` の前の root 実行部へ移動**。`\"` エスケープの
  二重引用符で生成し `chown "$CUSER"` する。これで**ネストが 1 段減り、シングルクォート不要**になる。
- 再発防止として実装仕様（`docs/impl/10_cli.md`）に「`-c` スクリプト内でシングルクォート使用禁止」の
  クォート制約を明記。
- 検証は「ホストシェル解釈 → コンテナ bash → su 内 bash」の 3 段を再現するハーネス（`docker`/`su`/
  `chown`/`claude` をスタブ化）で、`-c` 引数の個数と生成 JSON の妥当性（jq）を確認。

## 一般化した教訓（今後どう活かすか）

- **シェルのクォートを 3 段ネストしない**（ホスト `'…'` → `su -c "…"` → 内側 `echo '…'`）。破綻すると
  文字列が裸で露出し、さらに `{A,B}` がブレース展開されて `-c` 引数がスクリプトごと分裂する、という
  静かで致命的なバグになる。
- **ファイル生成のようなリテラル出力は、なるべく最外（root）で行い、後から chown で所有権を合わせる**。
  ネスト段数そのものを設計で減らすのが最も確実な対策。
- ネストしたシェル起動の検証は、目視でなく **`docker inspect` の `Cmd`（＝実際に渡った引数配列）**と、
  生成物のパーサ（jq 等）で機械的に行う。「`-c` に渡った引数が想定個数か」が破綻検知の急所。
- 壊れやすい抽出方式の検証ハーネスは常設せず、代わりに**クォート制約をドキュメントに明記**して
  再発を防ぐ、という割り切りも妥当。
