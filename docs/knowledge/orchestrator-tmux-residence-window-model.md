## 状況

コントローラを「端末を閉じても作業継続」させる常駐の器として、(A) setsid 完全デーモン化と
(B) tmux 常駐のどちらを採るか、また worker/壁打ち/介入を別セッションに置くか単一セッション配下の
ウィンドウに置くかを検討した。当初は別セッション方式で、生存判定に `has-session` を使っていた。

## 判断

- **常駐は tmux 常駐（Model B）を採用**（ユーザ判断）。コントローラ＋ダッシュボードを 1 プロセスの
  まま `orch-<CNAME>-main` セッション内で回し、tmux サーバを常駐の器にする。完全デーモン化で必要
  になる「ダッシュボード別プロセス化＋ファイル IPC」を回避でき、端末破壊耐性という本質要件は満たせる
  （tmux server 死という稀事象は状態からの resume で吸収）。
- **別セッション → 唯一のセッション配下のウィンドウにぶら下げる**方式へ変更。切替 `select-window`、
  生成 `new-window`、終了 `kill-window`、保持 `remain-on-exit`（dashboard=off／worker・wallbounce=on）。
  完了した worker のウィンドウは閉じ、メインは常駐。
- **生存判定を `has-session` → コントローラ実プロセス（pgrep）へ**修正。`has-session` は「空き殻
  セッション」を生存と誤検出する（`display-message` も同様の落とし穴があり `list-windows` 厳密照合へ）。
  空き殻（has-session=真だが pgrep=不在）は kill して起こし直し resume する。
- 巨大 instruction/prompt は `send-keys` で流し込まず **sidecar ファイル＋`$(cat)`** で読む
  （多 KB 文字列と引用符事故を回避）。パスは単一引用符エスケープ、生成物は機械所有ディレクトリ限定。
  定期セッション復旧は 5 秒スロットル。launcher スクリプトは `unset SLACK_BOT_TOKEN`
  （Slack 発信はコントローラに一本化）。

## 一般化した教訓（今後どう活かすか）

- **常駐要件（端末破壊耐性）を満たす手段として、まず tmux 常駐を検討する**。完全デーモン化は
  UI の別プロセス化と IPC を呼び込みコストが高い。本質要件が「器の永続」なら tmux で十分なことが多い。
- **tmux の生存判定に `has-session`/`display-message` を使わない**。セッションは残っても中身の
  プロセスが死ぬ「空き殻」を誤検出する。判定は常に「目的のプロセスが生きているか（pgrep）」で行う。
- 子作業は独立セッションに散らすより**単一セッション配下のウィンドウ**にぶら下げると、切替・可視化・
  完了時クローズ・一括管理が単純になる。`remain-on-exit` の on/off で「完了で消す/残す」を出し分ける。
- tmux へ長い文字列を送るときは `send-keys` を避け、ファイル経由（`$(cat)`）で渡す。
