## 状況

コンテナへ SSH 鍵を渡すのに、従来はホストの環境 agent（全鍵を保持しうる）を丸ごと転送していた。
「ディレクトリごとに ssh-add する鍵を変えたい」「他プロジェクトの鍵はコンテナから見えないように
したい」という要望に対応する必要があった。

## 判断

- **プロジェクト直下 `.claude-dev.yaml`（`ssh_keys:`）で鍵を解決**し、**プロジェクト専用 ssh-agent**
  （`~/.claude-dev/agents/<name>.sock`）を起動/再利用して**解決した鍵だけ**を登録・転送する。別
  プロジェクトの鍵はコンテナから見えない（隔離）。**秘密鍵ファイルは非マウント**（不変条件維持）、
  転送は agent ソケットのみ。むしろ環境 agent 丸ごと転送より**最小権限が向上**。
- 既登録鍵は**指紋照合でスキップ**（パスフレーズ再入力回避・冪等）。鍵 0 件時は SSH 転送しない。
  ソケット置き場は `chmod 700`（他ローカルユーザーの接続防止・多層防御）、実体は ssh-agent が 0600。
- macOS 版（方式D: 専用 agent＋socat TCP ブリッジ）も**プロジェクトごとに別 agent・別ポート**へ分離。
  `stop` は当該プロジェクトのブリッジのみ停止（agent は鍵保持のため残置）。
- **既知の制限は「見送り＋理由付き」で明示**。Unix ソケットパス長（約 108 byte）制限で深い `$HOME` だと
  `ssh-agent -a` が失敗し得るが、その場合は**警告して SSH 転送なしで継続（クラッシュしない）**。一般的な
  `$HOME` では問題にならないため名前ハッシュ化等は見送り。
- 最終的にユーザ要望で**グローバル `~/.config/claude-dev.yaml` へのフォールバック・自動生成・対話選択を
  廃止し、ローカル `.claude-dev.yaml` の `ssh_keys` のみを見る方式へ簡素化**した。

## 一般化した教訓（今後どう活かすか）

- 認証情報の転送は**「全部渡す」でなく「そのコンテキストに必要なものだけ」**を既定にする。環境 agent
  丸ごと転送は楽だが過剰権限。**プロジェクト専用 agent に指定鍵のみ**を載せると隔離と最小権限を両立できる。
- 秘密鍵は**ファイルとしてマウントせず agent ソケット経由**、という不変条件を OS が変わっても守る。
- 冪等性は**指紋照合でスキップ**して担保する（パスフレーズ再入力の UX 悪化を避ける）。
- **OS 由来の下限（ソケットパス長など）は握り潰さず、警告して安全側に縮退（転送なしで継続）**させる。
  一般ケースで踏まないなら過剰な回避策は入れず、「制限・理由・縮退挙動」をドキュメントに残す。
- **設定の解決経路は増やすほど複雑化する**。グローバル/ローカルの多段フォールバック＋自動生成＋対話選択は、
  ユーザにとって「どこが効いているか」不明瞭になりがちで、ローカル 1 箇所へ簡素化する判断もあり得る。
