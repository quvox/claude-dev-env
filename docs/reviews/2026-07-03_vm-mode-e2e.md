# VM モード（QEMU+virtiofs）実機 E2E 動作確認

日付: 2026-07-03
対象: VM モード実装（[docs/08_vm-mode.md](../08_vm-mode.md) / [docs/impl/80_vm-mode.md](../impl/80_vm-mode.md)）
方法: `claude-dev-claude` base に VM パッケージ＋スクリプトを重ねたテストイメージを、`--device /dev/kvm` 付きコンテナで起動し、`vm-up.sh` を実行してゲスト VM をブート。実 `/dev/kvm` のあるホストで実施。

## 実機検証で発見・修正した不具合

1. **virtiofsd が PATH に無い**（`/usr/libexec/virtiofsd`）→ `vm-up.sh` を絶対パス解決（`command -v virtiofsd || /usr/libexec/virtiofsd`）に修正。※ビルド確認段階で検出。
2. **ゲスト dockerd の tcp 待受設定が効かない**（当初 `--host=unix://…,tcp://127.0.0.1:2375` を `systemctl enable --now` で適用）:
   - `enable --now` は既起動の dockerd を再起動しないため override が反映されず。→ `systemctl restart docker` に修正。
   - unix を明示 `--host=unix://` すると `docker.socket` の fd と競合。→ `-H fd://` に修正（socket 活性化の unix を維持）。
   - tcp を `127.0.0.1` にするとホスト転送（hostfwd）が届かない（user-mode はゲストの SLIRP IP 宛）。→ **`-H tcp://0.0.0.0:2375`** に修正（露出は claude コンテナの 127.0.0.1 hostfwd 経由のみ）。

## 所見（4観点）
- 要件合致: bind mount がライブ反映付きで成立し（R1/R2）、claude 側から `DOCKER_HOST` で透過利用（R3）、claude コンテナは非 privileged（R4）を実機で確認。
- 無駄処理: cloud image はキャッシュし再 provision で再 DL しない。冪等（dockerd 到達時は no-op）。
- 処理時間: 初回は cloud image DL（~700MB）＋ゲスト provision で数分。以降のブートは短い。
- セキュリティ: claude コンテナは `--device /dev/kvm` のみ（非 privileged）。ゲスト dockerd は 0.0.0.0 待受だが到達は hostfwd（コンテナ 127.0.0.1）経由のみ＝ネットワーク非公開。
